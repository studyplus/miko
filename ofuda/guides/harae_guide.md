# 攻撃的検証ガイド

このファイルは攻撃的検証の手順書である。

- **初回モード（サブエージェント並列）:** 各サブエージェントは自分の担当セクション（A, B, C）を読み、その指示に正確に従うこと。
- **再実行モード（メインセッション）:** 全セクション（A, B, C）を通して読み、差分探索の参考にすること。

## 共通ルール

- **コードは読まない。** 実装マッピングはルール間の関係を理解する補助として参照してよいが、指摘はルール定義の問題に限定する。「ルール本文と実装の不一致」は祓えの対象外とする
- 指摘は「問題」だけを報告する。「問題ないこと」は報告しない
- 現実的に発生しうるなら、確信がなくても指摘する
- 各指摘は以下の構造で記述する:

```
### {短いタイトル}
- **検証軸**: (6軸のいずれか)
- **関連ルール**: RULE-XX, RULE-YY
- **深刻度**: 高 / 中
  - 高: ビジネスに直接的な損害、またはルールの矛盾でどちらを適用すべきか不明
  - 中: 曖昧さや穴があり、一定の技術力があるエンジニアでも正しい解釈にたどり着けない
  - （低と判断したものは報告しない。低 = 理論上の問題だが現実的な発生可能性が低い、または一定の技術力があるエンジニアなら迷わず正しく処理できる）
- **指摘**: 何が問題か（1〜2文）
- **詳細**: 具体的な状況の説明
- **対処案**: こうすれば解決する（思いつく場合）
```

- メインセッションには指摘リストのみ返す。指摘がなければ「指摘なし」と返す

### 自己レビュー（指摘を書いたら必ず実施）

指摘を書いた後、以下のフィルタを順に適用する。不合格なら削除する。

1. **現実性フィルタ** — この状況は現実的に発生しうるか？ 理論上の可能性だけなら削除
2. **驚きフィルタ** — 一定の技術力があるエンジニアがルール一覧を読んで、この指摘の状況を「言われなくても正しく処理できる」なら削除。言われないと間違える、または判断に迷うものだけ残す
3. **ビジネスルール固有性フィルタ** — ビジネスルールの定義の問題か？ 一般的な技術課題や実装設計の問題なら削除

### 指摘にしないもの

- 単一ルールの内容確認（「ORD-01 は在庫確認を求めている」→ それはルールの説明であって指摘ではない）
- 一般的な技術課題（「分散システムでは整合性が難しい」→ ビジネスルール固有の問題ではない）
- 「〜のルールがない」という指摘で、それが自明に不要な場合（「draft 状態の注文に対する配達完了通知のルールがない」→ そもそもありえない操作）
- 一定の技術力があるエンジニアなら迷わず正しく実装できる曖昧さ（「24時間以内」の境界値が未定義でも、実装者は「以下」か「未満」のどちらかを選んでテストを書く — ビジネス上どちらでも問題ない場合は指摘しない）
- 現実的な発生可能性が低い理論上の問題

## サブエージェント A: 論理的整合性

担当軸: **内部矛盾** + **不完全性**

### 内部矛盾

ルール同士が論理的に矛盾していないかを検査する。

**1. スコープ重複の検出**

全ルールの適用条件（どの操作、どの状態、どのアクターに適用されるか）を整理し、適用条件が重なるルールのペアを列挙する。各ペアについて、両方が同時に発火したとき結論が矛盾しないか確認する。

> 例: ORD-02「出荷済みの注文はユーザー操作ではキャンセルできない」と ORD-04「猶予期間経過後のキャンセルは管理者操作でのみ可能」— 管理者が出荷済み注文をキャンセルしようとした場合、ORD-02 は「出荷済みはキャンセル不可」と読めるが、ORD-04 は「管理者なら可能」と読める。ORD-02 の「ユーザー操作では」が管理者を除外しているかどうかで解釈が分かれる。

> 指摘しない例（矛盾ではないもの）: ORD-01「在庫引き当て済みでなければ確定できない」と PAY-01「決済失敗は注文確定をロールバックしない」— 適用タイミングが異なる（確定時 vs 確定後）ので矛盾しない。これは指摘しない。

**2. 優先順位の検査**

複数ルールが適用候補になるケースを特定し、どちらが優先されるかが明記されているか確認する。

> 例: ORD-03「猶予期間は24時間」と ORD-04「猶予期間後は管理者のみ」— ちょうど24時間時点でユーザーがキャンセル操作をした場合、ORD-03 と ORD-04 のどちらが適用されるか不明。「24時間以内」が「24時間ちょうどを含む」のか「含まない」のかで ORD-03 が適用される側か ORD-04 が適用される側か変わる。（これは境界の曖昧さにも該当するが、優先順位の観点からも指摘の価値がある）

**3. 死んだルールの検出**

他のルールの条件が常に先に評価され、決して発火しないルールがないか確認する。

> 例: もし「全注文に対してキャンセル猶予期間中は自由にキャンセルできる（ORD-03）」に加えて「confirmed 状態の注文は即座にキャンセルできる」というルールがあった場合、confirmed から猶予期間が始まるので、後者は ORD-03 に完全に包含されて独立した意味を持たない。

### 不完全性

ルール体系にカバーされていないケースがないかを検査する。

**1. 操作 x 状態マトリクス**

business_rules.md の「操作の定義」から全操作を、「状態遷移」から全状態を列挙する。全ての (操作, 状態) の組み合わせについて、適用されるルールがあるか確認する。状態遷移がなければスキップする。

作成するマトリクスのイメージ:

| | draft | confirmed | paid | shipped | completed | cancelled |
|---|---|---|---|---|---|---|
| 注文確定 | ORD-01 | - | - | - | - | - |
| 注文キャンセル | ? | ORD-03/04 | ORD-03/04, PAY-03 | ORD-02 | ? | - |
| 決済完了通知 | ? | (通常フロー) | - | - | - | ? |
| 配達完了通知 | - | - | - | (通常フロー) | - | - |
| 未決済自動キャンセル | - | PAY-02 | - | - | - | - |

`?` のセルがルール未定義の穴の候補。ただし、ありえない組み合わせ（draft に配達完了通知）は除外する。

> 例: 「cancelled 状態の注文に対する決済完了通知」が穴になりうる。PAY-01 で決済を非同期リトライしている間に注文がキャンセルされた場合、その後に決済が成功したらどうなるか？ どのルールもこのケースをカバーしていない。

> 例: 「completed 状態の注文に対するキャンセル」— ORD-02 は「出荷済み」を禁止しているが、「完了済み」については言及がない。完了後のキャンセルも同様に禁止されるべきだが、ルール上は穴。

**2. アクター x 操作マトリクス**

「操作の定義」のグループ（ユーザー操作、管理者操作、バッチ等）を確認し、各操作について全アクターからのアクセスが考慮されているか確認する。

> 例: 「注文作成」「注文確定」「注文キャンセル」はユーザー操作として定義されている。管理者が代理で注文を作成するケースは想定されているか？ 管理者代理操作にもキャンセル猶予期間（ORD-03）は適用されるのか？

> 例: 「未決済注文の自動キャンセル」はバッチ処理として定義されている。管理者が手動で同じ操作（未決済注文の強制キャンセル）を行う場合のルールはあるか？

**3. 境界のスコープ**

「関連ケイパビリティとの境界」を確認し、境界で「借りているもの」「渡しているもの」それぞれについて、失敗時や遅延時のルールが定義されているか確認する。

> 例: 在庫管理との境界で「在庫引き当てにはタイムアウトがある」「バッチによる自動キャンセルまで引き当てが残り続ける」と記述されている。在庫引き当てがタイムアウトした場合（在庫管理側でタイムアウト解放、注文管理側ではまだ confirmed）、注文側のステータスはどうなるか？ この状態不整合を扱うルールがない。

> 例: 配送管理との境界で「通知遅延により実際の配達完了とステータス更新にタイムラグが生じる」と記述されている。このタイムラグ中にユーザーがキャンセルを試みた場合（shipped 状態だが実際には配達済み）、ORD-02 により返品プロセスへ誘導されるが、実際にはまだ「出荷済み」として扱われている。これは意図通りか？

## サブエージェント B: 精度と堅牢性

担当軸: **境界の曖昧さ** + **時間軸の破綻**

### 境界の曖昧さ

ルールの条件が曖昧で、実装者の解釈次第で結果が変わらないかを検査する。

**1. 境界値の未定義**

数値条件（期間、回数、金額等）を持つルールを全て抽出し、各条件について「ちょうどその値」の場合の扱いが明記されているか確認する。「以上/以下」「未満/超」「含む/含まない」が曖昧なものを指摘する。

> 例: ORD-03「キャンセル猶予期間は注文確定から24時間」— 「24時間」は「24時間以内」なのか「24時間未満」なのか。確定時刻からちょうど24時間00分00秒の時点でキャンセルリクエストが来た場合の扱いが不明。

> 例: PAY-02「確定から24時間以内に決済が完了しない注文は自動キャンセル対象」— ORD-03 と同じ「24時間」を使っているが、こちらは明示的に「以内」と書いてある。ORD-03 には「以内」がない。同じ意味で書いているのか、意図的な差異なのか不明。

**2. 暗黙の前提**

各ルールが前提としている条件を洗い出し、その前提が business_rules.md 内で明文化されているか確認する。

> 例: ORD-02「出荷済みの注文はユーザー操作ではキャンセルできない。返品プロセスへ誘導する」— 「返品プロセス」の存在を前提としているが、返品プロセスが未定義。返品ケイパビリティが別に存在するのか、注文管理の一部なのか不明。

> 例: PAY-01「非同期リトライで整合性を回復する」— リトライの上限回数や最終的に失敗した場合の扱いが前提として暗黙。リトライが永久に続くのか、上限に達したらどうなるのか。

> 例: ORD-04「猶予期間経過後のキャンセルは管理者操作でのみ可能」— 「管理者」の定義が暗黙。全管理者がキャンセル権限を持つのか、特定の権限レベルが必要なのか。

**3. 用語の揺れ**

business_rules.md 内で同じ概念を指すのに異なる言葉が使われていないか確認する。用語集に定義があるか確認し、定義なく複数の表現が混在しているものを指摘する。

> 例: ORD-02 では「ユーザー操作」、ORD-04 では「管理者操作」と対比しているが、「操作の定義」では「ユーザー操作（API）」「イベント駆動」「バッチ処理」というグループ分けをしている。「ユーザー操作」と「ユーザー操作（API）」は同じ範囲を指すのか。

> 例: ORD-03 は「キャンセル猶予期間」を定義し、用語集にもあるが、ORD-04 は「猶予期間経過後」と略称を使っている。この文書内では問題ないが、別のケイパビリティに「猶予期間」という同名の異なる概念がある場合に混乱する。glossary.md を確認し、用語の衝突がないか確認する。

### 時間軸の破綻

時間経過や状態遷移の中でルールが壊れないかを検査する。

**1. 遷移の順序依存**

状態遷移図がある場合、同じ最終状態に到達する複数の経路を列挙し、各経路でルールの適用結果が同じになるか確認する。

> 例: cancelled に到達する経路は3つある:（a）confirmed → cancelled（猶予期間内キャンセル）、（b）paid → cancelled（出荷前キャンセル）、（c）confirmed → cancelled（PAY-02 自動キャンセル）。経路 (b) では PAY-03 により返金処理が自動開始されるが、経路 (a) と (c) では発動しない。これは意図通りだが、経路によって cancelled の意味合いが異なる（返金あり/なし）。cancelled 状態に返金状態を示す属性が必要ではないか。

**2. 並行操作**

同じエンティティに対して同時に実行されうる操作のペアを考え、両方が同時に走ったときルールが一貫した結果を保証するか確認する。

> 例: ユーザーがキャンセル操作を実行中（ORD-03 の猶予期間内）に、PAY-02 のバッチが同じ注文を自動キャンセル対象として処理する場合。両方が同時に cancelled への遷移を試みるが、二重キャンセルの扱いについてルールがない。

> 例: 決済プロバイダからの決済完了通知（confirmed → paid）と、ユーザーのキャンセル操作（confirmed → cancelled）が同時に到着した場合。confirmed 状態から paid と cancelled のどちらに遷移すべきかの優先順位がない。

**3. 既存データとの整合性**

ルールが追加・変更された場合、変更前に作られた既存データに新ルールを適用すると矛盾しないか考える。

> 例: ORD-03 の猶予期間が「24時間」から「12時間」に変更された場合、変更時点で確定から13〜24時間が経過している注文はどうなるか。旧ルールでは猶予期間内だったものが、新ルールでは期間外になる。変更の遡及適用についてルールがない。

> 例: PAY-02 の自動キャンセル条件が変更された場合、変更前に「自動キャンセル対象外」だった未決済注文が、変更後のバッチ実行で突然キャンセルされる可能性がある。

## サブエージェント C: ビジネスインパクト

担当軸: **ビジネス毀損** + **悪用耐性**

### ビジネス毀損

ルール通りに動いても、ビジネス的におかしな結果にならないかを検査する。

**1. ルール組み合わせの帰結チェック**

複数のルールを組み合わせて適用した結果が、ビジネス的に妥当か確認する。個々のルールは正しいのに、組み合わせると直感に反する結果になるケースを探す。

**手順（体系的に検査する）:**

1. 全ルールについて、以下を整理する:
   - **前提**: このルールが適用されるために真でなければならない条件
   - **効果**: このルールが適用された結果、何が真になるか（状態変更、値の決定、制約の発動）

2. ルールのペアを順に検査する（全組み合わせ）:
   - **効果→前提の衝突**: ルール X の効果が、ルール Y の前提を崩さないか？
   - **効果の合成**: ルール X の効果とルール Y の効果を同時に適用した結果、ビジネス的に妥当か？

3. 問題が見つかったペアについて、指摘として記述する

整理のイメージ（注文管理サンプル）:

| ルール | 前提 | 効果 |
|--------|------|------|
| ORD-01 | 注文確定操作 | 在庫が引き当て済みでなければ確定不可 |
| ORD-02 | キャンセル操作 + shipped 状態 | キャンセル不可、返品へ誘導 |
| ORD-03 | キャンセル操作 | 確定から24時間を猶予期間と定義 |
| ORD-04 | キャンセル操作 + 猶予期間外 | 管理者のみキャンセル可能 |
| PAY-02 | confirmed 状態 + 24時間経過 | 自動キャンセル |
| PAY-03 | 決済済み注文のキャンセル | 返金を自動開始 |

この表から、例えば PAY-02 の効果（自動キャンセル）と PAY-03 の前提（決済済み注文のキャンセル）を突き合わせると:「PAY-02 で自動キャンセルされる注文は confirmed（未決済）なので PAY-03 は発火しない」と確認できる。これは問題なし。

一方、もし別のケイパビリティに「サービス利用状態が active であることを前提とする」ルールがあり、同期ルールの効果で「特定条件で inactive に変更する」場合、その組み合わせで意図しないサービス停止が起きないか検査できる。

**ルール数が多い場合:** 全ペアの検査が現実的でなければ、以下の優先順で絞る:
- 異なるカテゴリ間のペア（同一カテゴリ内は設計時に考慮済みの可能性が高い）
- [導出] ルールを含むペア（導出の効果が他のルールの前提に波及しやすい）
- 境界セクションで言及されている外部ケイパビリティに関わるペア

> 例（効果→前提の衝突）: ORD-03 [導出]「猶予期間は24時間」→ PAY-02 [導出]「24時間以内に決済未完了なら自動キャンセル」。猶予期間の終了と自動キャンセルの発動が同時刻になるため、猶予期間ぎりぎりでユーザーがキャンセルしようとした瞬間にバッチが先に自動キャンセルする可能性がある。

> 例（効果→前提の衝突）: 「月間利用量が0のユーザーは inactive に分類」と「自動バッチ処理はユーザー操作としてカウントしない」— バッチの効果（操作としてカウントしない）が、分類ルールの前提（利用量で判定）に波及し、実態としてアクティブなユーザーが inactive に分類される。

> 例（効果の合成）: 「返品された注文は売上から除外」と「部分返品は返品として処理」— 両方の効果を合成すると、10個中1個だけ返品しても注文全体が売上から除外される。

> 例（効果の合成）: 「無料トライアルは30日間」と「紹介ボーナスで30日延長」— 両方適用すると60日間無料。ルール通りだがビジネス的に想定外。

> 例（効果の合成）: もし「返金額は注文時の単価で計算」[導出] と「クーポン適用後の価格で請求」[導出] が別々に定義されていた場合、返金額がクーポン適用前の金額になり、支払額より多く返金される可能性がある。

**2. ステークホルダー視点の回転**

ルールに関わるステークホルダー（ユーザー、管理者、運営、パートナー等）を特定し、各ステークホルダーの視点でルールの結果が合理的か確認する。

> 例（ユーザー視点）: PAY-01 により決済が非同期処理されるため、注文確定直後は「確定済みだが決済未完了」の状態。ユーザーには「注文完了」と見せているが、24時間以内に決済が通らなければ自動キャンセルされる。ユーザーは注文できたと思っているのに翌日キャンセルされるという体験は合理的か。

> 例（運営視点）: ORD-04 で管理者のみがキャンセル可能な注文について、管理者が不在の場合（深夜・休日）、猶予期間後のキャンセル依頼にタイムリーに対応できない。運営体制とルールが整合しているか。

> 例（配送パートナー視点）: PAY-03 で決済済み注文のキャンセル時に返金が自動開始されるが、キャンセルのタイミングによっては配送業者が既に集荷・配送準備を始めている可能性がある。配送コストの負担についてルールがない。

**3. 極端なケースでの妥当性**

件数・金額・期間が極端な場合にルールが妥当か確認する。「ゼロ」「1」「大量」「最長/最短」のケースを想定する。

> 例（ゼロ）: 0円の注文（全額クーポン適用等）の場合、PAY-02 の「24時間以内に決済が完了しない」はどうなるか。0円なら決済処理自体が不要だが、決済完了通知が来ないため PAY-02 で自動キャンセルされうる。

> 例（大量）: ある時間帯に大量の注文が集中し、バッチの PAY-02 実行時に数千件が同時に自動キャンセル対象になった場合、PAY-03 の返金処理も数千件同時に発火する。ルール上は正しいが、決済プロバイダへの返金リクエスト集中はビジネスリスク。

> 例（最短）: 注文確定直後（1秒以内）に決済完了通知が到着した場合でも、ORD-03 のキャンセル猶予期間は開始されるのか。決済済みなのに24時間キャンセル可能だと、実質的に「決済後も24時間は注文を取り消せる」ことになる。

### 悪用耐性

ルールの隙をついて不正な利益を得られないかを検査する。

**1. ルールの逆読み**

各ルールを「悪意のあるユーザー」の視点で読む。ルールの条件を意図的に満たす/回避する方法がないか考える。

> 例: ORD-01「在庫引き当て済みでなければ確定できない」+ ORD-03「猶予期間は24時間」→ 悪意のあるユーザーが人気商品を注文確定して在庫を24時間ロックし、他のユーザーの購入を妨害できる。キャンセル後に再度注文すれば無限に繰り返せる。

> 例: ORD-04「猶予期間後は管理者のみキャンセル可能」→ ユーザーが管理者にキャンセルを依頼することで、PAY-03 の自動返金を意図的に発動させ、商品受け取り前の返金を繰り返す。

**2. 操作の順序操作**

操作の順番を変えるだけでルールを回避できないか考える。本来想定されていない順序で操作した場合にルールが適切に機能するか。

> 例: ユーザーが注文を確定（confirmed）→ 猶予期間内にキャンセル（cancelled）→ 同じ商品を再度注文確定（新しい注文で confirmed）。2回目の注文にも猶予期間が適用されるため、事実上48時間分の在庫ロックが可能。これを繰り返すと任意の期間在庫をロックできる。

> 例: 決済完了通知の到着前にキャンセルし（confirmed → cancelled）、その後に決済完了通知が到着する。キャンセル済みの注文に対する決済成功の扱いがないため、決済は完了したが注文はキャンセル済みという状態が発生する。返金が必要だがPAY-03 は「決済済み注文のキャンセル時」にしか言及しておらず、「キャンセル済み注文への決済完了」は想定外。

**3. 境界の悪用**

数値条件の閾値ぎりぎりを狙った操作がないか考える。閾値を繰り返し利用して累積的な利益を得られないか。

> 例: ORD-03 の猶予期間を利用して、23時間59分でキャンセル → 即座に再注文 → また23時間59分でキャンセル、を繰り返す。毎回在庫を約24時間ロックしつつ、自分は一切決済しない。在庫ロック攻撃のコストがゼロ。

> 例: もし「月間キャンセル回数に制限なし」であれば、上記の在庫ロック攻撃や、セール価格での確定→通常価格に戻った後もキャンセル猶予で保持→気が変わったらキャンセル、という価格ヘッジが可能になる。キャンセル回数の上限についてルールがないことを確認する。
