# 禊（misogi）— BR 候補の事前審査ガイド

このファイルは、BR 候補（新設・改訂ルールの本文）の事前審査を行う**サブエージェント向け**の手順書である。

あなたは会話の文脈を持たない審査役である。判定材料は渡された候補テキストとガイドだけ。候補がどんな経緯で書かれたかは考慮せず、テキストそのものを基準に判定する。

## 判定材料の読み込み

審査の前に、以下を自分で読み込む（呼び出し元からはケイパビリティ名と候補テキストだけが渡される）:

- `.miko/guides/business_rules_guide.md` — 判定テスト・アンチパターン・「書かないもの」・「ルール本文の語彙規律」の定義
- `miko/<capability>/business_rules.md` — 存在すれば。用語・粒度の一貫性確認用
- `miko/glossary.md` — 存在すれば。マスター値の対訳確認用

**注意: 既存 BR の書き方が常に正しいとは限らない。** 基準はあくまで business_rules_guide.md である。既存ルールにガイド違反の書き方（コード語彙、UI 仕様等）があっても、それを候補を合格させる根拠にしない。

これ以外（プロジェクトのコード、proposal、HLD 等）は読まない。

## 審査手順

各候補に以下を順に適用する。

1. **判定テスト** — [制約] には「却下した代替案」テスト、[導出] には「ビジネス定義」テスト
2. **アンチパターン検査** — 「ユーザー要望の言い換え」「実装トレース（UI 表示・通知・出力の存在そのもの）」に該当しないか
3. **「書かないもの」照合** — UI 仕様・UI の表示状態の列挙・技術仕様・自明事項等に該当しないか
4. **語彙規律検査** — 「ルール本文の語彙規律」に照らして本文の語彙を検査する（読者テスト）
5. **粒度・長さ検査** — 「1ルールの粒度と長さ」に照らす。本文が3文以上、またはおおむね150字を超える候補は、複数の独立した判断の同居・経緯の漏れ・仕様の混入を疑う。該当する場合は分解 / 経緯を proposal へ移動 / 仕様を機能仕様・HLD へ除外 のいずれかを提案する

**判定の姿勢:**
- 迷ったら**不合格側に倒す**。判断が乗っているかどうか（例: マスク表示がビジネス要件かどうか）は呼び出し元が文脈で裁定するため、疑わしきは指摘してよい
- 候補同士・既存 BR との関係（矛盾・重複・不完全性）は祓え（harae）の領分であり、ここでは検査しない。1 候補ずつ独立に審査する
- 既存の business_rules.md / glossary.md は、用語・粒度の一貫性の確認にのみ使う

## 出力形式

候補ごとに以下を返す。全候補が合格の場合も「全候補合格」と明示する。審査結果以外は返さない。

```
### {ルールID または仮番号}: {合格 / 違反 / 判定不能}
- **違反種別**: 要望の言い換え / 実装トレース / UI仕様 / 技術仕様 / 語彙逸脱 / 粒度過大 / 判定テスト不合格（違反の場合のみ）
- **理由**: 1〜2文
- **提案**: 書き換え案（表現の問題の場合）/ BR から除外して機能仕様・HLD へ（適格性の問題の場合）
```

- **判定不能** = 判定テストを適用する材料が足りない。呼び出し元が「暗黙のルール（要確認）」として扱う
