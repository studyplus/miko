# business_rules.md 生成指示書

## このドキュメントの目的

機能のビジネスルールを `business_rules.md` として生成するための指示書。
既存の実装からの逆引き、および提案からの新規作成の両方に対応する。

---

## ドキュメント体系

### miko/ の整理軸

- `miko/` の下は **ケイパビリティ単位**（機能ではない）
- ケイパビリティ = 「注文管理」「在庫管理」「通知配信」のような大きな単位
- 機能 = 「注文キャンセル」「在庫引き当て」のような個別の操作。ケイパビリティの一部

### ケイパビリティごとの成果物

| ファイル | 役割 | メンテ |
|---|---|---|
| `business_rules.md` | ドメインの判断基準。この手法の主役 | AI がメンテ |
| `high_level_design.md` | ケイパビリティの構造と全体像 | AI がメンテ |
| `proposals/` | ケイパビリティへの変更提案と経緯 | 人が元ネタを出し、AI と相談しながら書く |

- `business_rules.md` と `high_level_design.md` は長期メンテ対象。実装と並行して AI が更新する
- ビジネスルールだけでは構造が見えず、ハイレベル設計だけではドメインの判断基準がわからない。両方あって初めてケイパビリティを理解できる

### システム全体のドキュメント

| ファイル | 役割 |
|---|---|
| `miko/system_high_level_design.md` | システム全体のアーキテクチャ（テナント構造、API 構造、コード探索ガイド等） |
| `miko/glossary.md` | 用語の定義。ケイパビリティごとに意味が異なる場合はスコープ付きで併記する |

### 使い捨ての成果物

| ドキュメント | 配置 | 目的 |
|---|---|---|
| speckit 成果物 | `specs/<feature>/` | Claude に徹底調査させる道具。git commit しない |

## 開発フロー

### 新規ケイパビリティの場合

```
/miko.new_cap <capability>
  → 対話しながら business_rules.md + high_level_design.md を作成

/miko.new_harae <capability>
  → 初回の攻撃的検証。harae.md を生成（推奨）

→ 指摘を修正したら /miko.harae <capability> で再検証をお勧めする
```

### 既存ケイパビリティに変更を加える場合

```
1. proposal を書く
   /miko.propose <capability>
   → 変更の背景・動機、ビジネスルールの変更、機能仕様を記述

2. proposal を検証する（推奨）
   /miko.harae <capability> <proposal>
   → proposal 適用後のルール体系を攻撃的に検証。指摘は proposal 内に記録される

3. speckit で仕様策定・実装
   /miko.speckit.specify → /miko.speckit.clarify → /miko.speckit.plan → /miko.speckit.tasks → /miko.speckit.analyze → /miko.speckit.implement
   → speckit の成果物は使い捨て。Claude に深く調査させるための道具
   → 大きな変更はフェーズ分割可能（親 proposal + サブ proposal）。各サブ proposal に対して実行する

4. 実装後にドキュメントを更新（自動）
   → business_rules.md を更新（proposal の「ビジネスルールの変更」を反映）
   → high_level_design.md を更新（構造に変更があった場合のみ）
   → harae.md を更新（proposal の「祓え検証」セクションがあれば適用）
```

**フェーズ分割:** 変更が大きい場合、`/miko.split_proposal` でプロポーザルを親（umbrella）+ サブにフェーズ分割できる。

### ドキュメントの流れ

```
proposal（人が元ネタを出し、AI と相談しながら書く）
    ↓
speckit（AI が使い捨て仕様を作って実装）
    ↓
business_rules.md（AI が更新：ドメインの判断基準）
high_level_design.md（AI が更新：構造の変化）
```

## ビジネスルールとは何か

**ビジネスルールとは、合理的な別の選択肢があったのに、ビジネスが選んだ判断のことである。** 「このビジネスでは何が真か」を記述するもので、「システムがどう振る舞うか」ではない。

条件には2種類ある。技術的に必然な制約（違反するとバグやデータ破壊が起きるもの）はビジネスルールではない。合理的な別の選択肢があったのにこちらを選んだもの — それがビジネスルールである。

### ビジネスルールと仕様の違い

ビジネスルールと仕様（機能仕様・技術仕様）は異なる。混同しないこと。

| | ビジネスルール | 仕様 |
|---|---|---|
| **問い** | 「何が真か」 | 「どう振る舞うか」 |
| **例** | 未決済の注文は出荷できない | キャンセルボタンを押すと確認ダイアログを表示する |
| **実装が変わったら** | 変わらない（ビジネスの判断だから） | 変わりうる（実現方法だから） |
| **書く場所** | business_rules.md | proposal の機能仕様セクション / spec.md |

**迷ったときの判定**: 「実装技術やUIを変えても、この記述は変わらないか？」— Yes ならビジネスルール、No なら仕様。

**粒度の原則**: 提案や議論が詳細であっても、BRの粒度は変わらない。UIの話・細かい仕様の話をしているときほど、このフィルタを強く意識すること。BRはドメインの判断基準であり、実装の設計書ではない。通常の機能追加で新設BRが5本を超えたら粒度が細かすぎる可能性が高い。

### 判定テスト

ルールを書く前に、以下のテストで「ビジネスルールかどうか」を機械的に判定する。

**[制約] の判定: 「却下した代替案」テスト**

そのルールに対して、合理的な代替案（別の選択肢）が存在するか？
- 代替案が書ける → ビジネスルールである。書く
- 代替案が「バグを作る」「データを壊す」しかない → ルールではない。書かない

例:
- 「再付与は冪等にする」→ 代替案「エラーにする」→ 書ける → **ルール**
- 「トランザクション内で作成する」→ 代替案はデータ不整合（バグ）→ **ルールではない**
- 「送信者は2種類」→ 現実的な代替案なし → **ルールではない**

**[導出] の判定: 「ビジネス定義」テスト**

技術的に必然ではなく、ビジネスが定義したマッピングか？
- Yes → ビジネスルールである。書く
- No → 書かない

**判定できない場合**: 「暗黙のルール（要確認）」として末尾に列挙し、人間に確認する。

### よくあるアンチパターン

判定テストは通したつもりでも、以下の 2 パターンは BR ではないので注意する。

判断軸は **「却下した代替案」テスト**。コードに書いてあるかどうかは判断軸にならない（BR は最終的にコードで表現されるので、突き詰めれば BR の多くはコードに表れる）。問うべきは「**そのルールは、別の選択肢がありえた中で選ばれたものか**」である。代替案が「やらない／そもそも書かない」しかないなら、それは判断ではなく実装の写経。

**アンチパターン1: ユーザー要望の言い換え**

ユーザーが「〜したい」「〜できるようにしたい」と言ったことを、そのまま「〜できる」という BR にしてはいけない。それは要望の言い換えであり、ドメインの判断ではない。要望の **背後にある条件・閾値・除外規則・対象範囲** だけが BR 候補になる。

| ユーザー発言 | BR にしてはいけない（要望の言い換え） | BR 候補になりうる（判断） |
|---|---|---|
| 「トライアル終了時に通知したい」 | 「トライアル終了時に通知する」 | 「終了の○日前に通知する」「再送は1回まで」 |
| 「管理画面でキャンセルできるようにしたい」 | 「管理画面でキャンセルできる」 | 「管理者は出荷後でもキャンセル可（一般ユーザーとは異なる）」 |
| 「未読リマインドを送りたい」 | 「未読リマインドを送る」 | 「未読の判定基準」「リマインド対象から除外する条件」 |

**アンチパターン2: 実装トレース系 — UI 表示・通知・出力の「存在そのもの」**

「画面に X を表示する」「Y のときメールを送る」「Z をログに記録する」のような、実装フローをそのままなぞった記述は BR ではない。**何を表示するか・何を送るかではなく、表示する条件・送る閾値・除外規則** が判断であり BR の本体。

| 実装トレース（BR にしない） | 判断（BR にする） |
|---|---|
| 「キャンセル時に確認ダイアログを表示する」 | （書かない。UI 判断であり、判断が乗っていない） |
| 「トライアル終了時にメール通知を送る」 | 「終了の **3日前** に予告通知。終了当日にも再送する」 |
| 「キャンセルを記録する」 | 「キャンセル後もレコードを残す（監査証跡のため）」 |

**注意**: アンチパターンに当たっても、その **背後に判断が乗っていれば BR**。「実装をなぞっただけか、代替案がありえた判断か」を見極める。迷ったら「却下した代替案」テストに戻り、**「やらない／別の条件にする」以外の合理的な代替案が書けるか** を厳しく問う。

### 重要な代替案の記録

判定テストの過程で重要な代替案を見つけた場合（「これを知らないと将来間違えそう」というもの）は、対応する proposal に「検討・却下した代替案」セクションとして追記する。business_rules.md にはルールだけ簡潔に書き、経緯は proposal に残す。

### 書かないもの

- **自明なこと**: 一定の技術力があるエンジニアなら知っていること
- **一般的な技術知識から導けること**: トランザクションでデータ整合性を保つ、等
- **このシステムの既存設計から推測できること**: 既存のアーキテクチャや慣習
- **実装方法**: どのクラス、どのメソッドで実現するか（実装マッピングに分離する）
- **UI仕様**: 表示形式、レイアウト、色、文言。**「画面に〜を表示する」「〜のときダイアログを出す」自体もBRではない**（判断が乗っていない実装トレース）
- **技術仕様**: データフォーマット、エンコーディング、通信プロトコル
- **ユーザー要望の単なる言い換え**: 「〜したい」を「〜できる」に書き換えただけのもの。要望の背後にある **条件・閾値・除外規則** だけが BR 候補
- **通知/送信/出力の「存在そのもの」**: 「X のときメールを送る」「Y を記録する」のような実装トレース。送る/送らないを分ける **条件**、タイミングの **閾値**、対象から **除外する規則** が BR

### 良い例・悪い例

| 悪い例（書いてはいけない） | 良い例（こう書く） | 理由 |
|---|---|---|
| フレームワークのバリデーション構文で一意制約をかける | 1ユーザーにつき、同一商品のレビューは1つしか投稿できない | フレームワーク非依存の記述 |
| レコードは物理削除せず論理削除する | 注文の監査証跡を保持するため、キャンセル後もレコードを残す | 「なぜ」が本当のルール |
| 定数定義でステータスを区別する | _(書かない)_ | 注文のステータスが複数あるのは自明 |
| トランザクション内で同時に作成する | _(書かない)_ | データ整合性の保ち方は技術判断 |
| 処理中は**全チャネル**で受付不可 | 処理中は、対象チャネルだけでなく全チャネルで新規リクエストを受け付けない | 「全チャネル」は非自明な判断 |
| 既にクーポンを持つユーザーに再付与したらエラーにせず上書きする | 既に有効なクーポンを持つユーザーへの再付与は冪等に処理する | エラーにする設計もありえる |
| キャンセル画面に確認ダイアログを表示する | _(書かない)_ | UI/コードを見れば自明 |
| トライアル終了時にメール通知を送る | トライアル終了の **3日前** に通知し、終了当日にも再送する | 送るかどうかではなく、**いつ・何回** が非自明な判断 |
| 管理者はオペレーター画面から注文をキャンセルできる | 管理者によるキャンセルは出荷後でも実行可能（一般ユーザーとの違い） | 操作の存在ではなく、**ルールの違い** がBR |
| 未読通知のリマインドを送る | 通知発行から **24時間後** に未読のものをリマインド対象とする。**3回送ったら停止** | 「送る」自体ではなく、**条件と打ち切り規則** がBR |

## ドキュメント構造

```
# {ケイパビリティ名} ビジネスルール

## 背景
## 状態遷移（該当する場合）
## 1. 操作の定義（Operations）
## 2. ビジネスルール・カタログ
## 3. 関連ケイパビリティとの境界
```

### 各セクションの指針

**背景**: なぜこの機能が必要になったか。ビジネス上の課題。2〜3段落以内。

**状態遷移**: 状態を持つ概念がある場合のみ。Mermaid の stateDiagram-v2 で図示し、遷移表で条件を補足する。

**操作の定義**: このケイパビリティに対する全操作を、アクター/トリガー別にグループ化して列挙する。ケイパビリティは複数の操作群の集合体であり、フラットなリストではなく構造で表現する。

例:
```markdown
### オペレーター操作（API）
- **紐づけ** — 説明。`POST /api/...`

### イベント駆動の自動同期
- **料金プラン変更** — 説明。

### 外部連携（外部サービス名）
- **AI権限の付与・剥奪** — 説明。

### バッチ処理
- **トライアル終了処理** — 説明。
```

**ビジネスルール・カタログ**: 後述の詳細ルールに従う。

**境界**: 他のケイパビリティとのビジネス上の責務分担。「借りているもの」「渡しているもの」「境界の注意点」の3軸。このケイパビリティの責務がどこまでかを明確にする。

> **注意:** アーキテクチャ上の決定事項（ADR）は business_rules.md には書かない。技術選択の「決定・意図・トレードオフ」は high_level_design.md の「設計上の特徴」セクションに記載する。

## ビジネスルール・カタログの書き方

### カテゴリ分け

関連するルールをカテゴリでグループ化する。カテゴリ名は短い日本語 + 英字プレフィックス。

```
### 権限管理（PERM）
### スレッド管理（THREAD）
```

### ルールID

`{プレフィックス}-{連番}` 形式。削除したルールの番号は欠番にする（詰めない）。harae.md や proposal がルールIDで指摘を参照しているため、番号を振り直すと参照が壊れる。

**最終採番の記録:** カテゴリの末尾に `<!-- last: PREF-03 -->` のように HTML コメントで最終採番を記録する。末尾のルールを削除した場合でも、次の採番番号がわかるようにするため。

### タグ

- **[制約]**: 「〜でなければならない」「〜は禁止」— 違反したらエラー
- **[導出]**: 「AならばBと定義する」— 事実から値や状態が決まる

迷ったら [制約] か [導出] のどちらかに寄せる。タグの厳密さより、ルールの内容が重要。

### 実装マッピング

ルール本体とは分離し、`<details>` タグで折りたたむ。

```markdown
**ルール:**
- **PERM-01** [制約] AI質問機能を利用するには、有効な権限が存在しなければならない。

<details><summary>実装マッピング</summary>

- PERM-01 → `XxxConcern#method` / 説明

</details>
```

1つのルールが複数箇所に実装されている場合は、同じルールIDで複数行を並べる。

```markdown
- ORD-02 → `Ec::Order#cancellable?` / ステータス判定
- ORD-02 → `Ec::OrdersController#cancel` / キャンセル前のガード
```

実装マッピングは既存実装から生成する場合にのみ記載する。提案段階では空でよい。

#### 表記規約

所属先（クラス、モジュール、ファイル、パッケージなど）を「スコープ」と呼び、以下の記号で参照先の種別を区別する。Ruby の RDoc 慣習に由来するが、miko では言語を問わず適用する。

| 種別 | 表記 | 意味 | 例 |
|------|------|------|----|
| スコープに属するもの | `Scope#name` | インスタンスメソッド等、スコープのインスタンスに紐づく | `Order#cancellable?` |
| スコープ自体に紐づくもの | `Scope.name` | static メソッド、モジュール関数等、インスタンスを介さずアクセスする | `OrderService.bulk_cancel` |
| 定数 / 型定義 | `Scope::NAME` | 定数、enum、型エイリアスなど | `Order::CANCEL_GRACE_PERIOD` |
| スコープ単位 | `Scope` | 特定メンバを指さない場合 | `ExpireUnpaidOrdersJob` |

「スコープ」は言語に応じてクラス、モジュール、ファイル、パッケージ、名前空間などに読み替える。スコープは名前空間を含む完全修飾名で記述する（例: `App::Models::Order#cancellable?`、`com.example.order.OrderService.bulkCancel`）。

## 生成モード別の追加指示

### モード A: 既存の実装から生成する場合

1. 対象のコードを読む（コントローラ、モデル、ジョブ、フォーム、ポリシー）
2. コードの振る舞いから「なぜそうなっているか」を抽出する
3. **コードの書き方ではなく、その背後にあるビジネス判断を書く**
4. 実装マッピングを `<details>` 内に記載する
5. コードに暗黙的に埋め込まれているが明文化されていないルールがあれば「暗黙のルール（要確認）」として末尾に列挙する

### モード B: 提案から生成する場合
* 提案には、ビジネスルールではない詳細な仕様が書かれることがあるが、それはビジネスルールには記載しない。

1. 提案の要件を読む
2. 要件をルール単位に分解する
3. 提案に明示されていないが論理的に必要なルールを推論し「暗黙のルール（要確認）」として末尾に列挙する
4. 提案に技術選択が含まれる場合、high_level_design.md の「設計上の特徴」への追記を検討する
