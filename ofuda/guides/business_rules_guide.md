# business_rules.md 生成指示書

## このドキュメントの目的

機能のビジネスルールを `business_rules.md` として生成するための指示書。
既存の実装からの逆引き、および提案からの新規作成の両方に対応する。

---

## ドキュメント体系

### miko/ の整理軸

- `miko/` の下は **ケイパビリティ単位**（機能ではない）
- ケイパビリティ = 「注文管理」「在庫管理」「通知配信」のような大きな単位
- 機能 = 「注文キャンセル」「在庫引き当て」のような個別の操作。ケイパビリティの一部

### ケイパビリティごとの成果物

| ファイル | 役割 | メンテ |
|---|---|---|
| `business_rules.md` | ドメインの判断基準。この手法の主役 | AI がメンテ |
| `high_level_design.md` | ケイパビリティの構造と全体像 | AI がメンテ |
| `proposals/` | ケイパビリティへの変更提案と経緯 | 人が元ネタを出し、AI と相談しながら書く |

- `business_rules.md` と `high_level_design.md` は長期メンテ対象。実装と並行して AI が更新する
- ビジネスルールだけでは構造が見えず、ハイレベル設計だけではドメインの判断基準がわからない。両方あって初めてケイパビリティを理解できる

### システム全体のドキュメント

| ファイル | 役割 |
|---|---|
| `miko/system_high_level_design.md` | システム全体のアーキテクチャ（テナント構造、API 構造、コード探索ガイド等） |
| `miko/glossary.md` | 用語の定義。ケイパビリティごとに意味が異なる場合はスコープ付きで併記する |

### 使い捨ての成果物

| ドキュメント | 配置 | 目的 |
|---|---|---|
| speckit 成果物 | `specs/<feature>/` | Claude に徹底調査させる道具。git commit しない |

## 開発フロー

### 新規ケイパビリティの場合

```
/miko.new_cap <capability>
  → 対話しながら business_rules.md + high_level_design.md を作成

/miko.new_harae <capability>
  → 初回の攻撃的検証。harae.md を生成（推奨）

→ 指摘を修正したら /miko.harae <capability> で再検証をお勧めする
```

### 既存ケイパビリティに変更を加える場合

```
1. proposal を書く
   /miko.propose <capability>
   → 変更の背景・動機、ビジネスルールの変更、機能仕様を記述

2. proposal を検証する（推奨）
   /miko.harae <capability> <proposal>
   → proposal 適用後のルール体系を攻撃的に検証。指摘は proposal 内に記録される

3. 実装する
   基本: /miko.quick_impl <capability | proposal>
   → speckit を通さず直接実装（単一の意図・既存構造の範囲内の変更が対象）

   重い変更（複数の意図が絡む、影響範囲の見極めに探索が要る、処理構造が大きく動く）は speckit フロー:
   /miko.speckit.specify → /miko.speckit.clarify → /miko.speckit.plan → /miko.speckit.tasks → /miko.speckit.analyze → /miko.speckit.implement
   → speckit の成果物は使い捨て。Claude に深く調査させるための道具
   → 大きな変更はフェーズ分割可能（親 proposal + サブ proposal）。各サブ proposal に対して実行する

4. 実装後にドキュメントを更新（自動）
   → business_rules.md を更新（proposal の「ビジネスルールの変更」を反映）
   → high_level_design.md を更新（構造に変更があった場合のみ）
   → harae.md を更新（proposal の「祓え検証」セクションがあれば適用）
```

**フェーズ分割:** 変更が大きい場合、`/miko.split_proposal` でプロポーザルを親（umbrella）+ サブにフェーズ分割できる。

### ドキュメントの流れ

```
proposal（人が元ネタを出し、AI と相談しながら書く）
    ↓
実装（基本は quick_impl で直接実装。重い変更は speckit が使い捨て仕様を作って実装）
    ↓
business_rules.md（AI が更新：ドメインの判断基準）
high_level_design.md（AI が更新：構造の変化）
```

## ビジネスルールとは何か

**ビジネスルールとは、合理的な別の選択肢があったのに、ビジネスが選んだ判断のことである。** 「このビジネスでは何が真か」を記述するもので、「システムがどう振る舞うか」ではない。

条件には2種類ある。技術的に必然な制約（違反するとバグやデータ破壊が起きるもの）はビジネスルールではない。合理的な別の選択肢があったのにこちらを選んだもの — それがビジネスルールである。

### ビジネスルールと仕様の違い

ビジネスルールと仕様（機能仕様・技術仕様）は異なる。混同しないこと。

| | ビジネスルール | 仕様 |
|---|---|---|
| **問い** | 「何が真か」 | 「どう振る舞うか」 |
| **例** | 未決済の注文は出荷できない | キャンセルボタンを押すと確認ダイアログを表示する |
| **実装が変わったら** | 変わらない（ビジネスの判断だから） | 変わりうる（実現方法だから） |
| **書く場所** | business_rules.md | proposal の機能仕様セクション / spec.md |

**迷ったときの判定**: 「実装技術やUIを変えても、この記述は変わらないか？」— Yes ならビジネスルール、No なら仕様。

**粒度の原則**: 提案や議論が詳細であっても、BRの粒度は変わらない。UIの話・細かい仕様の話をしているときほど、このフィルタを強く意識すること。BRはドメインの判断基準であり、実装の設計書ではない。通常の機能追加で新設BRが5本を超えたら粒度が細かすぎる可能性が高い。

### ドメインビジネスルール / システムビジネスルールの区別

ビジネスルールには **業務寄りのもの** と **システム寄りのもの** がある。両者を混在させず、明確に分離する。

| | ドメインビジネスルール | システムビジネスルール |
|---|---|---|
| **問い** | 紙の運用でも成り立つか | システム前提があってはじめて成り立つか |
| **例** | 出荷後はキャンセル不可 / 猶予期間は 24 時間 | 再付与は冪等にする / 応答失敗時は非同期リトライで整合性回復 |
| **典型語** | 期限・閾値・権限・対象範囲・除外規則 | 非同期・リトライ・冪等性・ロック・整合性回復 |

**判定: 「紙運用」テスト**

そのルールは、システムが存在しない（紙台帳と人手だけで運用する）世界でも意味として成立するか？

- Yes → **ドメインビジネスルール**
- No（システムの存在を前提とする概念に依存する）→ **システムビジネスルール**

**注意:** 既存のビジネスルールはコードから抽出されるため、表現がコードの語彙に引きずられがちである。「自動キャンセル」「即時反映」のような語は、紙運用なら「日次でキャンセル」「翌朝確認」と置き換え可能で、意味としては紙運用でも成立する → **ドメインビジネスルール**。一方、「非同期で整合性を回復する」「冪等に処理する」は紙運用に対応する概念がない → **システムビジネスルール**。コードの表現を一旦剥がしてから判定する（「ルール本文の語彙規律」と同じ filter）。

**1 つのルールに混在させない:** 1 ルールの中にドメイン判断とシステム判断が混ざっている場合は分解する。

- 悪: 「決済プロバイダの応答失敗は注文確定をロールバックしない（システム）。確定から 24 時間以内に決済が完了しない注文は自動キャンセル対象とする（ドメイン）」
- 良: 上記を 2 ルールに分割し、ドメインとシステムそれぞれのカテゴリに配置する

**カタログ上の配置:** カタログを `### 2.1 ドメインビジネスルール` と `### 2.2 システムビジネスルール` の 2 セクションに分け、各カテゴリはどちらか一方に置く。ドメインとシステムが混じるカテゴリはカテゴリの分割を検討する。

### 判定テスト

ルールを書く前に、以下のテストで「ビジネスルールかどうか」を機械的に判定する。

**[制約] の判定: 「却下した代替案」テスト**

そのルールに対して、合理的な代替案（別の選択肢）が存在するか？
- 代替案が書ける → ビジネスルールである。書く
- 代替案が「バグを作る」「データを壊す」しかない → ルールではない。書かない

例:
- 「再付与は冪等にする」→ 代替案「エラーにする」→ 書ける → **ルール**
- 「トランザクション内で作成する」→ 代替案はデータ不整合（バグ）→ **ルールではない**
- 「送信者は2種類」→ 現実的な代替案なし → **ルールではない**

**[導出] の判定: 「ビジネス定義」テスト**

技術的に必然ではなく、ビジネスが定義したマッピングか？
- Yes → ビジネスルールである。書く
- No → 書かない

**判定できない場合**: 「暗黙のルール（要確認）」として末尾に列挙し、人間に確認する。

判定テストを通過したルールは、本文の書き方を「ルール本文の語彙規律」（後述）に従わせる。

### よくあるアンチパターン

判定テストは通したつもりでも、以下の 2 パターンは BR ではないので注意する。

判断軸は **「却下した代替案」テスト**。コードに書いてあるかどうかは判断軸にならない（BR は最終的にコードで表現されるので、突き詰めれば BR の多くはコードに表れる）。問うべきは「**そのルールは、別の選択肢がありえた中で選ばれたものか**」である。代替案が「やらない／そもそも書かない」しかないなら、それは判断ではなく実装の写経。

**アンチパターン1: ユーザー要望の言い換え**

ユーザーが「〜したい」「〜できるようにしたい」と言ったことを、そのまま「〜できる」という BR にしてはいけない。それは要望の言い換えであり、ドメインの判断ではない。要望の **背後にある条件・閾値・除外規則・対象範囲** だけが BR 候補になる。

| ユーザー発言 | BR にしてはいけない（要望の言い換え） | BR 候補になりうる（判断） |
|---|---|---|
| 「トライアル終了時に通知したい」 | 「トライアル終了時に通知する」 | 「終了の○日前に通知する」「再送は1回まで」 |
| 「管理画面でキャンセルできるようにしたい」 | 「管理画面でキャンセルできる」 | 「管理者は出荷後でもキャンセル可（一般ユーザーとは異なる）」 |
| 「未読リマインドを送りたい」 | 「未読リマインドを送る」 | 「未読の判定基準」「リマインド対象から除外する条件」 |

**アンチパターン2: 実装トレース系 — UI 表示・通知・出力の「存在そのもの」**

「画面に X を表示する」「Y のときメールを送る」「Z をログに記録する」のような、実装フローをそのままなぞった記述は BR ではない。**何を表示するか・何を送るかではなく、表示する条件・送る閾値・除外規則** が判断であり BR の本体。

| 実装トレース（BR にしない） | 判断（BR にする） |
|---|---|
| 「キャンセル時に確認ダイアログを表示する」 | （書かない。UI 判断であり、判断が乗っていない） |
| 「トライアル終了時にメール通知を送る」 | 「終了の **3日前** に予告通知。終了当日にも再送する」 |
| 「キャンセルを記録する」 | 「キャンセル後もレコードを残す（監査証跡のため）」 |

**注意**: アンチパターンに当たっても、その **背後に判断が乗っていれば BR**。「実装をなぞっただけか、代替案がありえた判断か」を見極める。迷ったら「却下した代替案」テストに戻り、**「やらない／別の条件にする」以外の合理的な代替案が書けるか** を厳しく問う。

### 重要な代替案の記録

判定テストの過程で重要な代替案を見つけた場合（「これを知らないと将来間違えそう」というもの）は、対応する proposal に「検討・却下した代替案」セクションとして追記する。business_rules.md にはルールだけ簡潔に書き、経緯は proposal に残す。

### 書かないもの

- **自明なこと**: 一定の技術力があるエンジニアなら知っていること
- **一般的な技術知識から導けること**: トランザクションでデータ整合性を保つ、等
- **このシステムの既存設計から推測できること**: 既存のアーキテクチャや慣習
- **実装方法**: どのクラス、どのメソッドで実現するか（実装マッピングに分離する）
- **UI仕様**: 表示形式、レイアウト、色、文言、メール件名。**「画面に〜を表示する」「〜のときダイアログを出す」自体もBRではない**（判断が乗っていない実装トレース）
- **UI の表示状態の列挙**: 「モーダルの状態は10種類」のような画面状態の定義。BR ではなく high_level_design.md か SDD の成果物に書く
- **技術仕様**: データフォーマット、CSV の列構成、エンコーディング、通信プロトコル
- **ユーザー要望の単なる言い換え**: 「〜したい」を「〜できる」に書き換えただけのもの。要望の背後にある **条件・閾値・除外規則** だけが BR 候補
- **通知/送信/出力の「存在そのもの」**: 「X のときメールを送る」「Y を記録する」のような実装トレース。送る/送らないを分ける **条件**、タイミングの **閾値**、対象から **除外する規則** が BR

例外として、**出力内容そのものがビジネス判断であるもの**（例: 「メールアドレスはマスクして表示する」「有償の利用者のみ集計に含める」）は BR に書く。ただしその場合も書くのは判断だけで、画面名・列名・レイアウトといった画面の語彙は使わない（「ルール本文の語彙規律」参照）。

### 良い例・悪い例

| 悪い例（書いてはいけない） | 良い例（こう書く） | 理由 |
|---|---|---|
| フレームワークのバリデーション構文で一意制約をかける | 1ユーザーにつき、同一商品のレビューは1つしか投稿できない | フレームワーク非依存の記述 |
| レコードは物理削除せず論理削除する | 注文の監査証跡を保持するため、キャンセル後もレコードを残す | 「なぜ」が本当のルール |
| 定数定義でステータスを区別する | _(書かない)_ | 注文のステータスが複数あるのは自明 |
| トランザクション内で同時に作成する | _(書かない)_ | データ整合性の保ち方は技術判断 |
| 処理中は**全チャネル**で受付不可 | 処理中は、対象チャネルだけでなく全チャネルで新規リクエストを受け付けない | 「全チャネル」は非自明な判断 |
| 既にクーポンを持つユーザーに再付与したらエラーにせず上書きする | 既に有効なクーポンを持つユーザーへの再付与は冪等に処理する | エラーにする設計もありえる |
| キャンセル画面に確認ダイアログを表示する | _(書かない)_ | UI/コードを見れば自明 |
| トライアル終了時にメール通知を送る | トライアル終了の **3日前** に通知し、終了当日にも再送する | 送るかどうかではなく、**いつ・何回** が非自明な判断 |
| 管理者はオペレーター画面から注文をキャンセルできる | 管理者によるキャンセルは出荷後でも実行可能（一般ユーザーとの違い） | 操作の存在ではなく、**ルールの違い** がBR |
| 未読通知のリマインドを送る | 通知発行から **24時間後** に未読のものをリマインド対象とする。**3回送ったら停止** | 「送る」自体ではなく、**条件と打ち切り規則** がBR |

### ルール本文の語彙規律

判定テストを通過したルールでも、本文がコードの語彙で書かれていると BR の価値が落ちる。本文の合否は**読者テスト**で判定する。

**読者テスト**: コードを読んだことのないドメインエキスパートが、この文を理解できるか？ — No なら内容ではなく語彙の問題。以下の規律に従って書き直す。

ルール本文はビジネスの言葉、実装マッピングはコードの言葉。レイヤーを混ぜない。

技術用語を使ってよいかは「**実装を別技術に置き換えても、その語は残るか**」で判定する（判定テストの「実装技術やUIを変えても、この記述は変わらないか？」と同じ物差し）。残らない語は実現手段の名前であり、残る語は外から観測できる約束を表す。

| 語彙 | ルール本文での扱い | 行き先 |
|---|---|---|
| クラス名・メソッド名・ジョブ名・コード上の変数名 | **禁止**。ドメイン概念の言葉で言い換える | 実装マッピング |
| 実現手段の固有名（Job、enqueue、コールバック、トランザクション等） | **禁止**。実装を別技術に置き換えても残る言葉で書く | 実装マッピング |
| 振る舞いの意味論を表す技術用語（非同期、リトライ、ロールバック、冪等等） | **可**。外から観測できる約束であり、判断の中身になりうる。曖昧な日常語（「自動で」等）に言い換えて精度を落とさない。一般に通じにくい語は glossary で定義 | — |
| API のパラメータ名・レスポンスのフィールド名 | **禁止**。ビジネス語で書く | 実装マッピング |
| テーブル名・カラム名 | **可**。ドメイン概念の識別子として使ってよい | — |
| マスター値・enum 値・センチネル値の生値 | **原則禁止**。glossary か状態遷移セクションで対訳を定義したビジネス用語で書く。初出に限り「ビジネス用語（値）」の併記可 | 対訳は glossary.md / 状態遷移、生値は実装マッピング |
| 閾値・期限・件数上限などの数値 | **書く**。これこそ BR の本体 | — |

書き換え例:

| 悪い例（コード語彙） | 良い例（ビジネス語彙） |
|---|---|
| `UserCoupon` を持つユーザーはアカウント統合できない | クーポンの紐づけを持つユーザーは統合できない |
| ダミーデータは `create_with_unknown_start` で作成する | 開始日が不明なキャンペーンとして登録できる |
| ステータスが `inviting` または `active` であれば `granted` と判定する | 招待中または利用中であれば、権限は付与済みと定義する |
| Job が再実行された場合、例外を発生させて中断する | 完了済みの処理の再実行はエラーとして中断する（黙って成功扱いにしない） |
| 入力された ID リストは受付時に重複を排除する（`.uniq`） | 同じ対象が重複指定された場合は 1 件として扱う |

**表示・出力に判断が乗っている場合も、書くのは判断だけ。** 画面名・パネル位置・列構成・ボタン・文言・件名といった画面の語彙は本文に使わない。

- 悪: 「停止中の利用者は集計バーの分母から除外するが、右パネルのカード表示は維持する」
- 良: 「停止中の利用者は集計の分母から除外する。ただし停止前の記録の参照は維持する」

### 1ルールの粒度と長さ

1つのルールは1つの判断を表す。判断とは「却下した代替案」テストを1回通せる単位であり、本文はそれを言い切る最小限の長さにする。文の数に固定の上限はない（良い BR はたいてい1〜2文だが、判断が1つに保たれていれば文の数は問わない）。代わりに、次の2つを「詰め込みすぎ」のサインとして見る。どちらかに当たったら黄信号:

- **本文がおおむね150字を超える**: 経緯・理由・仕様が本文に紛れ込んでいる可能性が高い
- **独立した判断が3つ以上並んでいる**: 1ルールに複数の判断が同居している（補足を1つ足した2判断までは許容。例: 「自動再連携しない。再招待は手動運用とする」）

黄信号が出たら次のいずれかで対処する。

- **分解**: 独立した判断が複数あるなら、別のルールIDに割る
- **経緯を落とす**: 「なぜそう決めたか」「却下した代替案」は本文に書かず、結論だけ残す
- **仕様を除外**: UI・通知・出力の手順をなぞった部分は BR ではない。機能仕様 / high_level_design.md へ移す

150字はハード上限ではなく、再点検を促す合図である。意味を切り詰めたり、1つの判断を不自然に2本へ割ったりするための数値ではない。短く言い切れているなら何字でも、何文でも構わない。

## ビジネスポリシー（ビジネス方針）とは何か

**ビジネスポリシー（ビジネス方針）とは、ケイパビリティに通底する価値判断・指針のことである。** ビジネスルールの背景にある考え方を明文化する層。ポリシーが個別のルールを機械的に「導く」わけではない（ルールが先にあり、後から方針が見えてくることもある）。両者は **整合する関係** であって、上下の導出関係ではない。

コードから抽出されるビジネスルールの中には、本来は方針であるような価値判断が埋め込まれて見えなくなりがちである。それを明文化して分離するための層がビジネスポリシーである。

### ビジネスルールとビジネスポリシーの違い

| | ビジネスポリシー | ビジネスルール |
|---|---|---|
| **抽象度** | 高い（指針・価値判断） | 低い（具体的な判定基準） |
| **判定** | 任意のケースで客観的に判定できない | 任意のケースで客観的に判定できる |
| **例** | ユーザーの自主的なキャンセル権を尊重する | 出荷前であればユーザー操作でキャンセル可 |
| **位置** | ルールに通底する指針 | ポリシーと整合する具体的な決まり |

### 判定テスト: 「客観性」テスト

そのルール候補は、任意のケースにおいて客観的に判定できるか？

- Yes（条件・閾値・対象が明確で、機械的に判定できる）→ **ビジネスルール**
- No（価値判断・優先順位の表明であり、ケースごとに人間の解釈を必要とする）→ **ビジネスポリシー**

例:
- 「キャンセル猶予期間は確定から 24 時間」→ 時間で機械的に判定可 → **ルール**
- 「ユーザーの予期せぬ損失を防ぐことを優先する」→ ケースごとに何が「予期せぬ損失」か解釈が要る → **ポリシー**
- 「監査証跡を保持する」→ どこまで・どう保持するかは個別判断 → **ポリシー**
- 「キャンセル後もレコードを 3 年保持する」→ 機械的に判定可 → **ルール**

### ポリシーの書き方

- `## ビジネスポリシー（ビジネス方針）` セクションに ID 付きで列挙する（プレフィックスは `POL`）。セクション末尾に `<!-- last: POL-XX -->` で最終採番を記録する
- 1 ポリシー = 1 つの価値判断。短く言い切る
- 数は少なく保つ。ケイパビリティあたり 3〜5 個が目安。多すぎる場合は、ルールに分解できないかを再点検する
- 客観的に判定できるなら、それはポリシーではなくルール。**ポリシー欄に閾値や具体条件を書かない**
- ルール → ポリシーの対応は機械的なリンクでは管理しない。個別ルールでポリシーとの関係を強調したい場合は、ルール本文中に散文で言及する（例:「監査証跡（POL-02）の方針に従い、…」）

### ポリシーとアンチパターン

- **ルールのまとめ書き**: 個別ルールを抽象化しただけのもの（「キャンセルの条件を明確にする」等）は方針ではない。価値判断・優先順位として書けないなら、それは元のルールに溶ける
- **自明な価値観**: 「ユーザーを大事にする」「データを失わない」のような、どの事業にも当てはまる言葉はポリシーにしない。**このケイパビリティ固有の判断軸** だけを書く

## ドキュメント構造

```
# {ケイパビリティ名} ビジネスルール

## 背景
## ビジネスポリシー（ビジネス方針）
## 状態遷移（該当する場合）
## 1. 操作の定義（Operations）
## 2. ビジネスルール・カタログ
### 2.1 ドメインビジネスルール
### 2.2 システムビジネスルール
## 3. 関連ケイパビリティとの境界
```

### 各セクションの指針

**背景**: なぜこの機能が必要になったか。ビジネス上の課題。2〜3段落以内。

**ビジネスポリシー（ビジネス方針）**: ケイパビリティを貫く価値判断。`POL-{連番}` 形式で列挙する。詳細は前述の「ビジネスポリシー（ビジネス方針）とは何か」を参照。3〜5 個程度を目安に少数に保つ。セクション末尾に `<!-- last: POL-XX -->` で最終採番を記録する（ビジネスルールと同じ方式）。

**状態遷移**: 状態を持つ概念がある場合のみ。Mermaid の stateDiagram-v2 で図示し、遷移表で条件を補足する。

**操作の定義**: このケイパビリティに対する全操作を、アクター/トリガー別にグループ化して列挙する。ケイパビリティは複数の操作群の集合体であり、フラットなリストではなく構造で表現する。

例:
```markdown
### オペレーター操作（API）
- **紐づけ** — 説明。`POST /api/...`

### イベント駆動の自動同期
- **料金プラン変更** — 説明。

### 外部連携（外部サービス名）
- **AI権限の付与・剥奪** — 説明。

### バッチ処理
- **トライアル終了処理** — 説明。
```

**ビジネスルール・カタログ**: `### 2.1 ドメインビジネスルール`（紙の運用でも成り立つもの）と `### 2.2 システムビジネスルール`（システム前提のもの）の 2 セクションに分けて配置する。同一カテゴリ（ORD、PAY 等）はどちらか一方に置く。ドメインとシステムが混じるカテゴリはカテゴリの分割を検討する。詳細は前述の「ドメインビジネスルール / システムビジネスルールの区別」と、後述の「ビジネスルール・カタログの書き方」を参照。

**境界**: 他のケイパビリティとのビジネス上の責務分担。「借りているもの」「渡しているもの」「境界の注意点」の3軸。このケイパビリティの責務がどこまでかを明確にする。

> **注意:** アーキテクチャ上の決定事項（ADR）は business_rules.md には書かない。技術選択の「決定・意図・トレードオフ」は high_level_design.md の「設計上の特徴」セクションに記載する。

## ビジネスルール・カタログの書き方

### カテゴリ分け

関連するルールをカテゴリでグループ化する。カテゴリ名は短い日本語 + 英字プレフィックス。

```
### 権限管理（PERM）
### スレッド管理（THREAD）
```

### ルールID

`{プレフィックス}-{連番}` 形式。削除したルールの番号は欠番にする（詰めない）。harae.md や proposal がルールIDで指摘を参照しているため、番号を振り直すと参照が壊れる。

**最終採番の記録:** カテゴリの末尾に `<!-- last: PREF-03 -->` のように HTML コメントで最終採番を記録する。末尾のルールを削除した場合でも、次の採番番号がわかるようにするため。

### タグ

- **[制約]**: 「〜でなければならない」「〜は禁止」— 違反したらエラー
- **[導出]**: 「AならばBと定義する」— 事実から値や状態が決まる

迷ったら [制約] か [導出] のどちらかに寄せる。タグの厳密さより、ルールの内容が重要。

### 実装マッピング

ルール本体とは分離し、`<details>` タグで折りたたむ。

```markdown
**ルール:**
- **PERM-01** [制約] AI質問機能を利用するには、有効な権限が存在しなければならない。

<details><summary>実装マッピング</summary>

- PERM-01 → `XxxConcern#method` / 説明

</details>
```

1つのルールが複数箇所に実装されている場合は、同じルールIDで複数行を並べる。

```markdown
- ORD-02 → `Ec::Order#cancellable?` / ステータス判定
- ORD-02 → `Ec::OrdersController#cancel` / キャンセル前のガード
```

実装マッピングは既存実装から生成する場合にのみ記載する。提案段階では空でよい。

#### 表記規約

所属先（クラス、モジュール、ファイル、パッケージなど）を「スコープ」と呼び、以下の記号で参照先の種別を区別する。Ruby の RDoc 慣習に由来するが、miko では言語を問わず適用する。

| 種別 | 表記 | 意味 | 例 |
|------|------|------|----|
| スコープに属するもの | `Scope#name` | インスタンスメソッド等、スコープのインスタンスに紐づく | `Order#cancellable?` |
| スコープ自体に紐づくもの | `Scope.name` | static メソッド、モジュール関数等、インスタンスを介さずアクセスする | `OrderService.bulk_cancel` |
| 定数 / 型定義 | `Scope::NAME` | 定数、enum、型エイリアスなど | `Order::CANCEL_GRACE_PERIOD` |
| スコープ単位 | `Scope` | 特定メンバを指さない場合 | `ExpireUnpaidOrdersJob` |

「スコープ」は言語に応じてクラス、モジュール、ファイル、パッケージ、名前空間などに読み替える。スコープは名前空間を含む完全修飾名で記述する（例: `App::Models::Order#cancellable?`、`com.example.order.OrderService.bulkCancel`）。

## 生成モード別の追加指示

### モード A: 既存の実装から生成する場合

1. 対象のコードを読む（コントローラ、モデル、ジョブ、フォーム、ポリシー）
2. コードの振る舞いから「なぜそうなっているか」を抽出する
3. **コードの書き方ではなく、その背後にあるビジネス判断を書く**
4. ルール本文を「ルール本文の語彙規律」で検査する。コードを読みながら書くため、クラス名・メソッド名・enum の生値が本文に漏れやすい
5. 各ルール候補に「客観性」テストを適用し、客観的に判定できないもの（価値判断・優先順位）はビジネスポリシー（ビジネス方針、`POL`）として分離する
6. 各ルール候補に「紙運用」テストを適用し、ドメインビジネスルール（2.1）とシステムビジネスルール（2.2）に振り分ける。1 ルール内に両方が混在している場合は分解する
7. 実装マッピングを `<details>` 内に記載する
8. コードに暗黙的に埋め込まれているが明文化されていないルール・ポリシーがあれば「暗黙のルール（要確認）」として末尾に列挙する

### モード B: 提案から生成する場合
* 提案には、ビジネスルールではない詳細な仕様が書かれることがあるが、それはビジネスルールには記載しない。

1. 提案の要件を読む
2. 要件をルール単位に分解する
3. 各候補に「客観性」テストを適用し、ビジネスポリシー（ビジネス方針）候補（`POL`）とルール候補を分ける
4. ルール候補に「紙運用」テストを適用し、ドメインビジネスルール（2.1）とシステムビジネスルール（2.2）に振り分ける
5. 提案に明示されていないが論理的に必要なルール・ポリシーを推論し「暗黙のルール（要確認）」として末尾に列挙する
6. 提案に技術選択が含まれる場合、high_level_design.md の「設計上の特徴」への追記を検討する
