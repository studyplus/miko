# 注文管理（OrderManagement）ハイレベルデザイン — サンプル

> このファイルは miko スキルの品質基準を示すサンプルです。
> 実際のプロジェクトのシステム設計ではありません。

## この機能は何か

> 注文の作成から完了までのライフサイクルを管理する。
> 決済プロバイダ・配送業者との非同期連携を含み、注文状態の一貫性を保証する。

注文管理は EC サイトの中核ケイパビリティ。ユーザーの購入意思（カート）をビジネス上の約束（注文）に変換し、決済・配送という外部プロセスと協調しながら完了まで導く。

このケイパビリティの責務は「注文のライフサイクル管理」であり、決済処理そのものや配送ロジスティクスは外部ケイパビリティの責務。注文管理は両者からのイベント（決済完了、配達完了）を受け取ってステータスを遷移させる。

---

## テナント境界との関係

> データは Shop（店舗）単位で隔離されている。
> 注文は必ず1つの Shop に属し、クエリは Shop スコープで実行される。

```
Shop（店舗）
  └─ Order（注文）
       ├─ OrderItem（注文明細）
       └─ PaymentRecord（決済プロバイダとの通信記録）
```

- 管理者は自分の Shop の注文のみ閲覧・操作可能
- 集計やレポートも Shop 単位で生成される

---

## 概念モデル

> カラム定義ではなく、概念の関係性を示す。

- **Order** — 注文のライフサイクルを管理する中心エンティティ。`draft → confirmed → paid → shipped → completed` の状態を持つ。`cancelled` は confirmed/paid のどちらからも遷移可能
- **OrderItem** — 注文に含まれる商品の明細。注文時点の単価をスナップショットとして保持する（商品マスタの価格変更に影響されない）
- **PaymentRecord** — 決済プロバイダとの通信を記録する。注文とは独立したライフサイクル（pending → success/failed）を持ち、1つの注文に複数回の決済試行が紐づく可能性がある

---

## 外部システムとの関係

> 注文管理は外部の決済・配送・在庫システムと非同期で協調する。
> 外部システムの障害が注文操作をブロックしない設計。

### 決済プロバイダ

- 注文確定時に決済リクエストを**非同期ジョブ**で送信する
- 決済完了は Webhook で受け取る（プル型ではなくプッシュ型）
- 決済プロバイダの応答遅延・障害は注文確定をブロックしない
- 決済失敗時はジョブのリトライで回復し、一定回数失敗で注文を自動キャンセルする

### 在庫管理

- 注文確定時に在庫を確保し、キャンセル時に解放する
- 在庫確保はタイムアウト付き。確保後に注文が放棄されると在庫が戻らないリスクがあり、バッチで期限切れ注文の在庫を解放する

### 配送管理

- 決済完了後に出荷依頼を送信する
- 配達完了は配送業者からの通知で受け取る
- 通知遅延により、実際の配達完了とステータス更新にタイムラグが生じる

## 処理の全体像

```
ユーザー操作
  注文作成 → draft 状態で保存
  注文確定 → 在庫確保 → confirmed に遷移 → 決済リクエスト（非同期）

外部イベント
  決済完了通知（Webhook）→ paid に遷移 → 出荷依頼
  配達完了通知 → completed に遷移

キャンセル
  ユーザーキャンセル → 猶予期間内かつ出荷前のみ → 在庫解放 → cancelled に遷移

バッチ処理
  未決済注文の期限切れ → 確定から24時間経過で自動キャンセル → 在庫解放
```

---

## 設計上の特徴

### 非同期決済による結果整合性
- **決定:** 注文確定と決済完了を分離し、決済を非同期で処理する
- **意図:** 決済プロバイダの応答時間や障害に依存せず、ユーザーの注文確定操作を即座に完了させる
- **トレードオフ:** 「確定したが決済されない」注文が一時的に存在する。バッチによる自動キャンセル（PAY-02）で回収するが、その間はユーザーに「注文完了」と見せている状態のため、UX 上の配慮が必要

### 単価スナップショット
- **決定:** 注文明細に注文時点の単価を保存する
- **意図:** 商品マスタの価格変更が過去の注文に影響しない
- **トレードオフ:** 価格修正を遡及適用したい場合に対応できない

## 関連ケイパビリティとの境界

> 他のケイパビリティとの責務分担を明示する。

- **決済管理（Payment）**: 借りるもの — 決済リクエストの送信・結果通知の受信 / 渡すもの — 注文 ID・金額・通貨 / 境界の注意点 — 決済プロバイダの選択・リトライポリシーは決済管理の責務。注文管理は結果（成功/失敗）だけを受け取る
- **在庫管理（Inventory）**: 借りるもの — 在庫確保・解放 API / 渡すもの — 商品 ID・数量 / 境界の注意点 — 在庫不足時の注文キャンセルは注文管理が判断するが、在庫数の管理自体は在庫管理の責務
- **配送管理（Shipping）**: 借りるもの — 配達完了通知 / 渡すもの — 出荷依頼（注文 ID・配送先） / 境界の注意点 — 配送業者の選択・追跡は配送管理の責務
