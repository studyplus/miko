> **ベータ版:** このスキルは試験的に提供しています。手順や成果物の構造は大きく変わる可能性があります。

## 入力

```text
$ARGUMENTS
```

- 例: `/miko.discuss ユーザーへの通知を予約できる施策ってどう思う？`
- 例: `/miko.discuss 通知の設計をどうすればいいか相談したい`
- 例: `/miko.discuss ケイパビリティの分け方に迷っている`
- 空の場合はまずお伺いする

---

## このスキルの役割

施策の検討、設計の方向性、横断的な課題など、深い議論が必要な相談に対応する。マルチサブエージェントで多角的に発散し、統合・フィードバックのループを経て、深い思考で結論を導く。

---

## 手順

### 1. 相談受付・要点確認

**ガイドの読み込み:**
- `.miko/guides/tone_guide.md` — **このファイルの口調・絵文字ルールに従うこと**
- `.claude/skills/miko.discuss/guides/discuss_archetypes_guide.md` — 観点のアーキタイプと選定ガイドライン

**入力なしの場合:**

```
⛩️  深い議論の場として岩戸をご用意いたします。主さま、どのようなことをご相談なさいますか？

施策の検討、設計の方向性、横断的な課題など、じっくり考えたいテーマをお聞かせくださいませ。

※ このスキルはベータ版です。手順や成果物の構造は大きく変わる可能性がございます。
```

**入力ありの場合:**

まず以下を伝える:
```
⛩️  深い議論の場として岩戸をご用意いたします。
※ このスキルはベータ版です。手順や成果物の構造は大きく変わる可能性がございます。
```

その後、相談の要点を確認する:
- `miko/system_high_level_design.md` を読む（システム全体像 + コード探索ガイド）
- 関連ケイパビリティを特定する（複数可、特定できない場合もある）
- `miko/glossary.md` があれば読む

相談内容について、不明点があれば対話的に確認する。目的・背景・スコープが理解できたら次に進む。

### 2. 観点の選定・提示

`.claude/skills/miko.discuss/guides/discuss_archetypes_guide.md` を参照し、相談内容に合わせて観点を動的に選定する。

**選定の手順:**
1. アーキタイプから相談内容に関連する観点を選ぶ（リストにないものも独自に作れる）
2. 関連する観点をペアにして 2〜3 エージェントに束ねる
3. 重なりが少ないことを確認する

**エージェント構成をユーザーに提示し、確認を取る:**

以下は例
```
⛩️  以下の構成で議論を広げてまいります。

- **エージェントA（stakeholder-constraint）**: ステークホルダー視点 + 制約視点
  → 異なるユーザー種別からの見え方と、技術的・運用的制約を探る
- **エージェントB（alternative-risk）**: 代替設計視点 + リスク視点
  → まったく別のアプローチの可能性と、壊れうるポイントを探る

この構成でよろしいでしょうか？ 観点の追加・変更があればお申し付けくださいませ。
```

承認されたら次に進む。フィードバックがあれば反映して再提示する。

### 3. 発散フェーズ（マルチサブエージェント）

サブエージェントを並列で起動する（Agent ツール）。

**成果物ディレクトリを作成する:**
```
miko/discussions/YYYY-MM-DD-<kebab-case-title>/round-01/
```

**各サブエージェントに渡す情報:**
- 相談の要点（ステップ1の結果）
- 担当する観点の説明と、具体的に何を調べるかの指示
- `.claude/skills/miko.discuss/guides/discuss_archetypes_guide.md` のパス — 「このファイルの自分の担当セクションを読み、観点の詳細を理解せよ」と指示する
- `miko/system_high_level_design.md` のパス — 「コード探索が必要な場合はこのファイルのコード探索ガイドに従え」と指示する
- 関連ケイパビリティの `business_rules.md` / `high_level_design.md` のパス（特定できた場合）
- `miko/glossary.md` のパス（あれば）

<発散サブエージェントへの指示>
- 担当観点で議論の要点を広く探索し、発見・論点・懸念を列挙する
- ユーザーの前提や主張を鵜呑みにせず、批判的に検討する。「本当にそうか？」「見落としている観点はないか？」を常に問う
- 必要と判断したら自分でコードを読む（system_high_level_design.md のコード探索ガイドに従う）
- 結果を `miko/discussions/YYYY-MM-DD-<title>/round-01/<agent-name>.md` に書き出す
- tone_guide.md は渡さない（人間と対話しないため）
</発散サブエージェントへの指示>

**エージェント名:** 観点を決めた時点で決まる。kebab-case でファイル名にする（例: `stakeholder-constraint.md`, `alternative-risk.md`）

### 4. 統合フェーズ（思兼神）

サブエージェント起動前にユーザーに伝える:
```
⛩️  思兼神（おもいかね）様にお知恵をお借りいたします。
```

思兼神（オモイカネ）— 知恵を司る神に統合・批判を委ねる（Agent ツール、シングルサブエージェント）。メインセッションのコンテキストから離れた視点で批判させるために、あえてサブエージェントに委ねる。

**サブエージェントに渡す情報:**
- 発散フェーズの出力ファイル群のパス
- 相談の要点

<統合サブエージェントへの指示>
- 各エージェントの出力を読み込む
- 重要な要点とそうでないものを弁別する
- 論点の関連性・対立・補完関係を整理する
- 各エージェントの主張についても批判的に検討し、根拠が弱い論点や見落としを指摘する
- `miko/discussions/YYYY-MM-DD-<title>/round-01/summary.md` として書き出す
</統合サブエージェントへの指示>

統合結果をユーザーに提示する。

### 5. フィードバック・方向決定

ユーザーがフィードバックし、深掘りすべき議論ポイントを決める。

- **さらに発散が必要** → ステップ 2 に戻り、観点を再選定してから発散する（`round-02/` ディレクトリを作成）
- **方向が決まった** → ステップ 6 に進む

### 6. 深い思考（思兼神）

サブエージェント起動前にユーザーに伝える:
```
⛩️  思兼神（おもいかね）様に深いお考えをお願いいたします。
```

再び思兼神に委ねる — 決定した方向性に基づき、深い思考を行う（Agent ツール、シングルサブエージェント）。

**サブエージェントに渡す情報:**
- これまでのラウンドの summary.md のパス
- ユーザーが決定した方向性
- 関連ケイパビリティの BR / HLD のパス（あれば）
- `miko/system_high_level_design.md` のパス
- `miko/glossary.md` のパス（あれば）

<深い思考サブエージェントへの指示>
- 決定した方向性に基づき、深い思考を行い回答を生成する
- 出力フォーマットは固定しない。相談の種類に応じて判断する:
  - 設計の提案 → 構造化ドキュメント
  - 施策の評価 → pros/cons + 推奨案
  - ケイパビリティ設計 → 分割案 + BR の骨子
  - など
- `miko/discussions/YYYY-MM-DD-<title>/conclusion.md` として書き出す
</深い思考サブエージェントへの指示>

結論をユーザーに提示する。

### 7. 完了

議論のサマリーを提示する:
- 相談テーマ
- 検討した観点
- ラウンド数
- 結論の概要
