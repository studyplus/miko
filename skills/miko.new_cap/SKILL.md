## 入力

```text
$ARGUMENTS
```

入力の形式: `<capability_name> <概要（自由記述、省略可）>`

- 例: `/miko.new_cap order_management 注文の作成・決済・キャンセルを管理する`
- 例: `/miko.new_cap notification`
- 空の場合はエラー: 「⛩️  ケイパビリティ名をお願いいたします（例: `/miko.new_cap order_management 注文管理`）」

---

## このスキルの役割

新規ケイパビリティの business_rules.md と high_level_design.md を **対話しながらゼロから定義する**。

人間の知識と AI の調査を組み合わせて正引きで作る。既存コードがあればそれを活用するが、なくても使える。

**出力:**
- `miko/<capability>/business_rules.md` — 初期ビジネスルール（実装マッピングは未決の場合あり）
- `miko/<capability>/high_level_design.md` — 初期ハイレベルデザイン

**catchup との違い:**

| 観点 | miko.new_cap | miko.catchup |
|------|-----------|---------------|
| 用途 | 新規作成 | 既存ドキュメントの更新 |
| 前提 | ドキュメントがまだない | ドキュメントが既にある |
| コード | あれば活用、なくてもOK | 既存コードから差分を検出 |
| 対話 | ヒアリング + 確認 3 箇所 | 確認 2 箇所 |

---

## 手順

### 1. 入力検証

- `$ARGUMENTS` の先頭ワードからケイパビリティ名を取得し、スネークケースに正規化する
- 残りの部分を概要として記録する

### 2. 既存ファイル確認

- `miko/<capability>/business_rules.md` または `miko/<capability>/high_level_design.md` が既に存在するか確認する
- **存在する場合はエラーとして中止する:**
  > ⛩️  `miko/<capability>/business_rules.md` が既に存在しております。
  > - 既存ドキュメントの更新は `/miko.catchup <capability>` をお使いください
  > - 既存ルールへの変更提案は `/miko.propose <capability>` をお使いください

### 3. ガイドと参考資料の読み込み

以下のファイルを **すべて** 読み込む。

**指示書（必須）:**
- `.miko/guides/business_rules_guide.md` — business_rules.md の生成ルール。判定テストの方法
- `.miko/guides/tone_guide.md` — ユーザーとの対話スタイル。**このファイルの口調・絵文字・出力言語ルールに従うこと**

**システム全体の文脈:**
- `miko/system_high_level_design.md` — システム全体のアーキテクチャ、**コード探索ガイド**（プロジェクトごとに異なる）
- `miko/glossary.md` — 用語集（存在する場合）

**実例（品質の基準）:**
- `.miko/examples/business_rules.md` — business_rules.md のサンプル
- `.miko/examples/high_level_design.md` — ケイパビリティ HLD のサンプル
- `.miko/examples/system_high_level_design.md` — システム HLD のサンプル。コード探索ガイドの書き方の参考

**プロジェクト固有の実例（あれば）:**
- `miko/` 配下に既存の business_rules.md や high_level_design.md があれば、それらも品質基準として参照する
- ただし既存 BR の書き方が常にガイドに適合しているとは限らない。ガイドと矛盾する書き方（コード語彙、UI 仕様等）は手本にしない。基準は常に `.miko/guides/business_rules_guide.md`

### 4. ケイパビリティのヒアリング

ケイパビリティの全体像を理解するために対話する。概要が十分に明確であればすぐに確認①に進む。

**ヒアリング観点:**
- **目的**: このケイパビリティは何を解決するか？ なぜ必要か？
- **主要な操作**: ユーザー（管理者/一般ユーザー/システム）は何ができるか？
- **データの核**: 中心になるデータ（テーブル/概念）は何か？
- **状態**: 状態遷移が存在するか？ ある場合はどんな状態があるか？
- **外部連携**: 他のシステムやケイパビリティとの接点は？
- **制約**: 知っている制約やビジネスルールは？（部分的でOK）

**注意:**
- 人間は全てを言語化できていないことが多い。「他に制約はありますか？」ではなく、具体的なシナリオを挙げて「この場合はどうなりますか？」と聞く
- 選択肢と推奨を提示する（「A と B がありえますが、〜の理由で A を推奨します」）
- 「わからない」「まだ決めていない」は正当な回答。未決事項として記録する

### 5. 【確認①】理解の確認

ヒアリング内容を「目的」「主要な操作」「中心データ」「状態遷移」「外部との接点」「既知の制約・ルール」「未決事項」にまとめて提示する。

加えて **ネガティブシナリオ確認** を含める:
- ヒアリングで明示されなかった境界条件を 3〜5 個、具体的な「〜は禁止ですか？許可ですか？」形式で提示する
- 以下の 5 分類を観点として使う:
  1. 境界ケース（ヒアリング内容から推測される限界値・端のケース）
  2. 操作の組み合わせによる衝突（同時操作、矛盾する操作の組み合わせ）
  3. 異常系・不正利用パターン
  4. 時間軸に関する境界（期限切れ、同時実行等）
  5. 権限・アクター間の境界
- 抽象的な「他にありますか？」ではなく、このケイパビリティ固有のシナリオを挙げる
- 回答から暗黙のルール候補を抽出する

ネガティブシナリオへのご回答もお願いする旨を明記する（「ネガティブシナリオへのご回答もお願いいたします。」等）。回答は次のステップで暗黙のルール候補を抽出するインプットとなるため、省略不可。

フィードバックを反映して再提示。承認されたら次に進む。

### 6. コード調査

承認された確認①に基づき、関連する既存コードと隣接ケイパビリティを探索する。

**6a. 既存コードの探索**

**探索対象:** `miko/system_high_level_design.md` の **コード探索ガイド** に従い、ケイパビリティに関連するファイルを探索する。

**探索対象外:** `.miko/` ディレクトリは miko の内部リソースであり、プロジェクトのコードではない。探索・精読の対象にしないこと。

**着目点チェックリスト（コード読解時に必ず確認する）:**
- [ ] **バリデーション**: モデル/フォームのバリデーション定義 → 制約ルールの候補
- [ ] **enum / 定数定義**: ステータスや種別の定義 → 状態遷移・導出ルールの候補
- [ ] **状態遷移のガード条件**: 状態変更前の条件チェック → 制約ルールの候補
- [ ] **エラーメッセージ**: エラー文言に埋め込まれたビジネス判断 → 暗黙のルール候補
- [ ] **ポリシー / 認可**: アクセス制御の条件 → 権限系の制約ルール候補
- [ ] **コールバック / フック**: before/after の処理 → 導出ルールや副作用の候補
- [ ] **スコープ / クエリ条件**: WHERE 句やスコープの条件 → 暗黙の前提条件の候補
- [ ] **外部 API 呼び出し**: 連携先との契約・制約 → 境界ルールの候補

**コードが見つかった場合:**
- 上記チェックリストに従ってコードを読み、ルール候補を追加抽出
- 実装マッピングの素材として記録

**コードが見つからなかった場合:**
- 完全に新規のケイパビリティ。実装マッピングは全て `（未実装）` とする

**6b. 隣接ケイパビリティの BR 読み込み**

- `miko/*/business_rules.md` を Glob で検出し、確認①の「外部との接点」に挙がったケイパビリティの business_rules.md を読む
- 外部との接点に該当しなくても、ケイパビリティ名やドメイン概念が関連しそうなものは読む
- **確認する観点:**
  - 同じ概念に矛盾する定義がないか（用語の不整合）
  - 境界の責務分担が既存 BR と整合するか
  - 隣接 BR から借りられるルール（重複定義を避ける）
- 隣接 BR との整合性の問題を発見した場合、ステップ 9 の確認②に「隣接 BR との整合性」セクションを追加して報告する

### 7. ルール策定・構造化

ヒアリング + コード調査の結果を統合し、business_rules.md と high_level_design.md の内容を策定する。

**business_rules.md の策定:**
- `.miko/guides/business_rules_guide.md` の判定テストを全候補に適用
- ルール ID を採番（新規ケイパビリティなので連番は 01 から開始）
- 実装マッピング:
  - 既存コードがあるルール → コードの場所を記載
  - 未実装のルール → `（未実装）` マークを付ける
- 判定できないものは「暗黙のルール（要確認）」に列挙

**⚠️ 粒度の原則: ヒアリングの詳細度に引きずられない**

ヒアリングではユーザーが UI・操作・要望（「〜したい」）を細かく語ることが多いが、それらをそのまま BR にしてはいけない。ガイドの判定テスト（却下した代替案 / ビジネス定義）に加え、ガイドの **「よくあるアンチパターン」** に当たっていないか必ず確認する。

- **「〜したい」「〜できるようにしたい」が出てきた場合**: 要望の言い換え（「〜できる」）は BR ではない。要望の背後にある **条件・閾値・除外規則・対象範囲** だけが BR 候補
- **「画面に〜を表示する」「〜のとき通知する」**: 実装をなぞっただけの記述は BR ではない。判断（**いつ・何回・どの条件で除外するか**）が乗っているもの **だけ** が BR
- **新規ケイパビリティの BR 数の目安**: 規模により幅はあるが、操作 1 つあたり BR が 3 本を超えたら粒度を疑う。各ルールに「却下した代替案」テストを再適用し、**「やらない／別の条件にする」以外の合理的な代替案が書けるか** を厳しく問う

判断軸は「却下した代替案」テストに集約する。コードに書いてあるかどうかは判断軸にならない（BR は最終的にコードで表現される）。コードに表れていても、別の選択肢がありえた判断であれば BR として残す。

候補から外したものは BR ではなく、HLD の構造説明や proposal/spec の機能仕様で表現する。「BR から外したから捨てる」ではなく「書く場所を変える」だけ。

**high_level_design.md の策定:**
- ケイパビリティの全体像を構造化
- 既存コードがあればクラス構成を記載、なければ `（設計未定）` とする

### 8. 禊 — ルール候補の事前審査（サブエージェント）

ルール候補をユーザーに提示する前に、サブエージェント（Agent ツール）を 1 体起動して事前審査する。起草した本人による自己検査はヒアリングの発言に引きずられるため、文脈を持たない審査役にテキストだけで判定させる。

**サブエージェントに渡す情報:**
- `.miko/guides/misogi_guide.md` のパス — 「このファイルを読み、指示に正確に従え」と指示する
- ケイパビリティ名（審査役はここから判定材料を自分で読み込む）
- 審査対象のルール候補テキスト（仮番号、[制約]/[導出] タグ付き。まだファイルに存在しないため本文を渡す）

**会話の経緯・ユーザーの発言は渡さない。** 文脈が審査を汚染するため、判定材料は候補テキストとガイドだけにする。

**審査結果の裁定（メインセッション）:**
- 違反が妥当 → 書き換え案を適用する。BR 不適格は候補から外し、HLD や proposal/spec の機能仕様に書く場所を変える
- 違反を覆す（出力内容そのものがビジネス判断である場合等）→ 覆す理由を明示し、本文は「判断だけを書く」形に整える
- 判定不能 → 「暗黙のルール（要確認）」へ

### 9. 【確認②】ルール候補の確認

禊の裁定を反映したルール候補を「採用/暗黙のルール・要確認/未決事項」に分類してユーザーに提示し、採否と扱いを確認する。禊の審査結果（審査件数・修正・除外・覆しと理由）も簡潔に報告する。

### 10. 【確認③】ハイレベルデザインの確認

HLD の骨子（`.miko/examples/high_level_design.md` と同等の構造）をユーザーに提示し、「この構造で生成してよろしいでしょうか？」と確認する。

概念モデルの各概念には **実装状態** を付記する:
- **既存** — 既存のテーブル/クラスをそのまま使う
- **新規予定** — この実装で新規作成する
- **設計未定** — 方針が未決定（未決事項として記録）

この分類は実装フェーズでの設計判断（既存テーブル利用 vs 新規作成等）に影響するため省略しない。

### 11. ファイル生成

確認②③のフィードバックを反映し、2 ファイルを生成する。

**business_rules.md の生成:**
- `.miko/guides/business_rules_guide.md` の「ドキュメント構造」に従う（モード A / モード B の混合）
- 実装マッピング:
  - 既存コードがある場合: `<details>` 内に記載
  - 未実装の場合: `<details>` 内に `（未実装）` マーク付きで記載
- 未決事項がある場合、末尾に「未決事項」セクションを追加

```markdown
## 未決事項

| 項目 | 状況 | 決定方法 |
|------|------|---------|
| 〜の扱い | 未定 | proposal で決定 |
```

**high_level_design.md の生成:**
- `.miko/examples/high_level_design.md` の構造・抽象度を基準にする
- テーブルのカラム定義、API エンドポイント一覧、クラス構成ツリーは **含めない**
- 未実装・設計未定の部分は `（設計未定）` マークで明示

**system_high_level_design.md の更新（必須）:**
- `miko/system_high_level_design.md` の「ケイパビリティ一覧」テーブルに、新しいケイパビリティの行を追加する
- 書式は既存の行に合わせる。例:
  ```
  | {ケイパビリティ名} | {概要（1文）} | `miko/{capability}/` |
  ```
- **このステップを忘れないこと** — business_rules.md と high_level_design.md の生成と同時に必ず実行する

**glossary.md の更新:**
- 新しい用語が出てきた場合は `miko/glossary.md` に追加する（ファイルがなければ作成）
- `.miko/examples/glossary.md` のフォーマットに従い、該当ケイパビリティのセクションに追加する
- 複数ケイパビリティで共通の用語は `## 全体` セクションに置く

**生成時の注意:**
- miko.catchup と同じ品質基準を目指す
- ルール本文に実装詳細を含めない
- 未決事項は隠さず明示する — 「決まっていないことが明確」は価値がある

### 12. 完了報告

生成したファイル、ルール総数（制約/導出の内訳、実装済み/未実装）、暗黙のルール数、未決事項数をサマリーテーブルで提示する。次のアクションとして `/miko.new_harae` によるルール検証をお勧めする。
