---

## 入力

```text
$ARGUMENTS
```

---

## このスキルの役割

speckit.implement のワークフローを基盤として実行し、以下を変更する:

1. **フェーズごとに止まる** — 各フェーズ完了後に人間の確認を挟む
2. **test_design.md の活用** — テスト作成時に合意済みテストケースを参照する
3. **機能ごとのセルフレビュー** — テスト完了後にセルフレビュー → /simplify
4. **最終フェーズの実行** — 機能横断リファクタリング
5. **デザインドキュメントの更新** — business_rules.md と high_level_design.md にプロポーザル・実装の変更を反映

---

## 手順

### 1. speckit.implement の実行（Miko 拡張）

speckit の `implement` の手順を以下の順で探索して読み込み、基盤として実行する:

1. `.claude/skills/speckit-implement/SKILL.md` — spec-kit の skills モード（`--ai claude` の現デフォルト）
2. `.claude/commands/speckit.implement.md` — commands モード（旧レイアウト）

どちらも存在しない場合はエラー中止する:
> ⛩️  speckit の `implement` スキルが見つかりません。`specify init --ai claude` を先にお願いいたします。

**以下の点を変更する:**

#### 追加の文脈読み込み

speckit.implement のステップ 3（Load context）で、以下のファイルも追加で読み込む:

- `test_design.md`（FEATURE_DIR 内） — テスト観点とテストケース一覧
- spec.md 内の「Miko コンテキスト」からケイパビリティ名を取得
- `miko/<capability>/business_rules.md` — ルール ID の参照用

#### フェーズごとの確認停止

speckit.implement のステップ 6（Execute implementation）を以下に置き換える:

**実行開始前に、タスク全体の規模を判断する:**

- tasks.md のフェーズ数とタスク数を確認する
- **シンプルな場合**（フェーズが2つ以下、かつタスク総数が10件以下）:
  - 「タスクがシンプルですので一気に実装してもよろしいでしょうか？フェーズごとにお止めすることもできます。」と確認する
  - 承認されたら全フェーズを連続実行する（最終フェーズの前では必ず止まる）
- **それ以外の場合**:
  - フェーズごとに実行・確認を繰り返す

**フェーズ完了時の確認:**

各フェーズの全タスクが完了したら、完了タスク・変更ファイル・次のフェーズの概要を提示して確認を取る。

- フィードバックがあれば反映してから次に進む
- **「戻って修正して」と言われた場合:** 該当フェーズ（または該当タスク）に戻り、指摘箇所を修正してから再度確認を取る。前のフェーズへの差し戻しも受け付ける

**最終フェーズの前では必ず止まる**（一気通貫モードでも）。

#### テスト作成時のルール

テストタスクを実行する際:

**test_design.md がある場合:**
- test_design.md のテストケース一覧を **チェックリスト** として使う
- 各テストケースに対応する spec を作成する
- テストケース一覧にないテストは原則追加しない
- 追加が必要と判断した場合は、理由をコメントで明記する

**test_design.md がない場合:**
- spec.md の Acceptance Scenarios からテストを生成する

**テスト除外ルール（常に適用）:**
- フレームワークが保証する振る舞い（validates のパラメータチェック等）のテストは書かない
- 親クラス・共通基盤が担保する処理（認可、認証）のテストは書かない

#### 機能ごとのセルフレビュー

各機能のテストフェーズが完了したら、次の機能に進む前に以下を実行する:

**1. セルフレビュー**

実装したコードを読み返し、以下の観点でチェックする:

- **責務の配置**: 各コードが、使用フレームワークの規約上正しいレイヤー/ファイルに置かれているか
- **命名**: メソッド名・変数名が、そのレイヤーの責務を正しく反映しているか（他レイヤーの用語を使っていないか）
- **フレームワークの機能活用**: フレームワークが提供する仕組みで実現できることを自前で実装していないか

**識別子をリネームした場合はリポジトリ全体を grep する**: クラス名・モジュール名・ファイルパス・関数名・定数名等を変更した場合、型チェッカーやテストでは捕捉できない参照（文字列リテラル、コメント、テストフィクスチャ名、設定ファイル、動的ロードのパス、`miko/` 配下の design docs・proposals・harae、README やその他 markdown、CI 設定など）が残る。`grep` でリポジトリ全体から旧名を洗い出し、ヒットごとに以下を判断する:

- 同名の別概念ではないか（語境界・文脈の一致）
- 過去の経緯記録（解決済み harae 項目・廃止 proposal・過去の commit log を引いた docs 等）として旧名のまま残すべきか
- リネーム結果が文脈上正しい表記か（複合語・派生語の扱い）

機械的な一括置換はしない。判断に迷うヒットは、置換せずに主さまに提示して確認する。

問題を発見したら、その場で修正する。

**2. /simplify 実行**

セルフレビュー完了後、`/simplify` を実行してコードの重複・品質・効率をチェック・修正する。

**注意:** 機能が1つしかない場合はこのステップをスキップし、最終フェーズで一括実行する。

#### 最終フェーズの実行

tasks.md の最終フェーズに到達したら、以下の順序で実行する:

**1. 機能横断のリファクタリング**
- 主さまにベースブランチ名を確認する（例: 「このブランチのベースブランチ（マージ先）を教えてくださいませ」）。自動推定はしない
- ベースブランチからの差分全体（`git diff <base-branch>...HEAD`）を確認する
- 機能間の重複コード整理、命名の統一、モデルの調整等を行う
- 変更がなければスキップ
- このステップで識別子をリネームした場合は、上記「機能ごとのセルフレビュー」の **識別子をリネームした場合はリポジトリ全体を grep する** ルールをこの差分全体に対しても適用する

#### デザインドキュメントの更新

最終フェーズ完了後、Miko コンテキストがある場合（プロポーザル経由で実装した場合）に実行する。

**1. ガイドの読み込み**
- `.miko/guides/business_rules_guide.md` — ルールの書き方、実装マッピングの形式
- `.miko/examples/business_rules.md` — 品質基準
- `.miko/examples/high_level_design.md` — 品質基準

**2. business_rules.md の更新**
- プロポーザルの新設・改訂・廃止をそのまま `miko/<capability>/business_rules.md` に適用する
- ルール本文はプロポーザルに書かれた内容をそのまま使う（実装中に変わった場合は実態に合わせる）
- 今回の実装で作成・変更したファイルを元に、該当ルールの実装マッピング（`<details>` 内）を更新する
- 既存ルールでも実装場所が変わったものがあれば更新する
- **spec.md の Miko コンテキストに複数ケイパビリティのサブプロポーザルが含まれている場合:** 各サブプロポーザルの BR 変更セクションに従って、各ケイパビリティの `miko/<capability>/business_rules.md` をそれぞれ更新する

**3. high_level_design.md の更新**
- 今回の実装で構造に変更があった場合、`miko/<capability>/high_level_design.md` を更新する
- 新しい概念モデル、処理フロー、外部連携の追加・変更を反映する
- 構造に変更がなければスキップ

**4. harae.md への転記**
- プロポーザルに「祓え検証」セクションがある場合、その内容を `miko/<capability>/harae.md` に適用する。harae.md のフォーマットは `.miko/guides/harae_format_guide.md` に従う
- **既存指摘のステータス変更**: 祓え検証に記載されたステータス変更を harae.md の該当エントリに反映する。指摘一覧テーブルのステータスセル更新・打ち消し線付与、詳細セクションのステータスと解決フィールドの更新を行う
- **新規指摘の追加**: 祓え検証の新規指摘を harae.md に追加する。番号は既存の最大番号の続きから採番する。ステータス・解決フィールドが省略されている場合は `open` / `—` として補完する
- 祓え検証セクションがなければスキップ

**5. 親プロポーザルの進捗更新**
- プロポーザルの先頭に `<sub-proposal@{親のmiko相対パス}>` マーカーがある場合（サブプロポーザルの場合）、`miko/{パス}` で親プロポーザルを開き、該当フェーズのチェックボックスにチェックを入れる（`- [ ]` → `- [x]`）
- **複数のサブプロポーザルが同フェーズにある場合:** すべてのサブに対して実装が完了しているときに、該当フェーズにチェックを入れる。親プロポーザルのフェーズ構成で同フェーズに列挙されているすべてのサブプロポーザルの実装を確認すること
- 該当しなければスキップ

**6. 確認**
- 更新内容を提示し、主さまの確認を取る

**7. 他ケイパビリティの実装マッピング更新（Sonnet サブエージェント）**

Sonnet サブエージェント（Agent ツール、model: "sonnet"）を起動して実行する。

サブエージェントに渡す情報:
- `.miko/guides/cross_capability_mapping_guide.md` のパス — 「このファイルを読み、指示に従え」と指示する
- メインケイパビリティ名
- ベースブランチ名（ステップ 1 で確認済みの値を使用）

#### それ以外の speckit.implement の手順

セットアップスクリプト実行、チェックリスト確認、進捗トラッキング、エラーハンドリング等は speckit.implement の手順をそのまま実行する。

### 2. 完了報告

完了タスク数、変更ファイル数、テスト数、セルフレビュー修正数、/simplify 修正数、BR 更新件数（複数ケイパビリティの場合はケイパビリティごと）、HLD 更新有無、harae 転記件数をサマリーテーブルで提示する。テストの実行を依頼する。
