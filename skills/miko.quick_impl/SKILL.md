---

## 入力

```text
$ARGUMENTS
```

入力の形式: 以下のいずれか

- proposal のパス（例: `/miko.quick_impl miko/order_management/proposals/2026-03-01-fix-trial-grace-period.md`）
- ケイパビリティ名（例: `/miko.quick_impl order_management` — 最新の proposal を自動選択）
- 変更指示の自由記述（例: `/miko.quick_impl OrderService を OrderFacade にリネーム`）
- 空の場合はエラー: 「⛩️  proposal のパス、ケイパビリティ名、または変更指示をお願いいたします（例: `/miko.quick_impl order_management`）」

---

## このスキルの役割

小規模な変更を speckit フロー（specify → clarify → plan → tasks → analyze → implement）をスキップして直接実装する。proposal がある場合は BR 変更を含めて、ない場合は変更指示から直接、実装とドキュメント追従まで行う。

miko の原則として BR ルール本文の変更には proposal が必要。proposal がない入力で BR 本文変更が必要になると判定された場合は propose に誘導する。

**対象外（フルフローまたは propose に誘導）:**
- 複数の意図が絡む変更
- 影響範囲の見極めに追加の調査・探索が必要な変更
- 既存の処理フローの構造が大きく動く変更
- BR ルール本文を変える変更で、それを認可する proposal がない

**迷ったらフルフローを推奨する。**

---

## 手順

### 1. 入力解決

`$ARGUMENTS` を以下のルールで解決する:

- `miko/` で始まり `.md` で終わるパス → そのパスを proposal として使用。パスから cap 名を抽出
- snake_case の単一トークンで `miko/<name>/` ディレクトリが存在 → `miko/<name>/proposals/` 内の最新ファイルを proposal として使用
- 上記いずれにも該当しない → `$ARGUMENTS` を変更指示として扱う（proposal なし）

**proposal があった場合の追加検査:**

- 親プロポーザル（umbrella proposal）の検出 — 先頭に `<umbrella-proposal>` マーカーがある場合、エラーとして中止する:
  > ⛩️  親プロポーザルは直接実装できません。サブプロポーザルを指定してください。
- 未分割プロポーザルの検出 — 先頭に `<needs-split>` マーカーがある場合、エラーとして中止する:
  > ⛩️  このプロポーザルは分割が必要です。
  > まず `/miko.split_proposal` でフェーズ分割してから、サブプロポーザルを指定してください。

### 2. 資料読み込み

**共通:**
- `.miko/guides/business_rules_guide.md` — ルールの書き方、実装マッピングの表記規約
- `.miko/guides/tone_guide.md` — 対話スタイル。**このファイルの口調・絵文字・出力言語ルールに従うこと**
- `miko/system_high_level_design.md` — コード探索ガイド
- `miko/glossary.md` — 用語集（存在する場合）

**proposal がある場合の追加:**
- `miko/<capability>/business_rules.md` — 現在のビジネスルール
- `miko/<capability>/high_level_design.md` — 現在の構造（あれば）
- proposal ファイル

**proposal がない場合:** 上記共通のみを先に読み込む。コード調査中に cap 固有の BR/HLD が必要になった時点で随時参照する（どの cap に触れるかは調査結果で決まる）。

### 3. コード調査

`miko/system_high_level_design.md` の **コード探索ガイド** に従い、変更に関連するコードを探索・精読する。

- proposal がある場合: proposal の変更対象コードを調査
- proposal がない場合: 変更指示の対象コードを調査

いずれの場合も、影響範囲（他ファイル・他モジュール・他 cap からの参照）を把握する。

**探索対象外:** `.miko/` ディレクトリは miko の内部リソースであり、プロジェクトのコードではない。探索・精読の対象にしないこと。

### 4. スコープガード

コード調査の結果を踏まえ、以下の 4 軸で判定する。いずれかに該当すれば中止し、適切なスキルに誘導する。

**(a) 意図の単一性** — 変更が単一の意図・目的で説明できるか。BR ルールが何本に分解されるかは問わない（例: 「キャンセル後通知」は 1 ルールでも 3 ルールでも意図は単一）。複数の関心事が絡む場合:

> ⛩️  この変更は複数の意図が絡んでおります。フルフローをお勧めいたします。

**(b) 影響範囲の明瞭さ** — 変更点から波及先がコード上素直に辿れるか。波及の見極めのために広範な探索（当たりをつけるためのプロトタイプ、仮説検証、多段の読み込み）を要する場合:

> ⛩️  この変更は影響範囲の見極めに追加の探索が必要です。フルフローをお勧めいたします。

**(c) 機械性または局所性** — 機械的な変換（リネーム、シグネチャ変更、参照の一括置換 等）か、あるいは局所的な変更で完結するか。既存の処理フローの構造が大きく動く場合:

> ⛩️  この変更は処理構造に大きく影響いたします。フルフローをお勧めいたします。

**(d) BR ルール本文の変更を伴う場合、それを認可する proposal があるか。** proposal がない、または proposal に記載のない BR 本文変更が必要な場合:

> ⛩️  この変更はビジネスルールに影響いたします。
> `/miko.propose <capability>` でプロポーザルをお作りしてからお進みくださいませ。

**注記:**
- ケイパビリティ境界を跨ぐこと自体は中止理由にしない。BR 本文を変えないリファクタリングなら、複数 cap の実装・実装マッピングを同時に動かしてよい
- ファイル数・行数は目安にすぎない。機械的なリファクタリングなら多くのファイルに及んでも対象内。上記の 4 軸で判定する

**対象と判定した場合、次に進む。**

### 5. 【確認】実装方針の提示

変更内容（ファイルごとの変更概要）、テスト方針、ドキュメント更新の見込み（BR 本文・実装マッピング・HLD・harae のうち、実際に触る予定のもの）を簡潔に提示し、確認を取る。

### 6. 実装 + テスト

確認された方針に従い、実装とテストを行う。

**実装後、セルフレビューを実行:**
- **責務の配置**: フレームワークの規約上正しいレイヤー/ファイルに置かれているか
- **命名**: メソッド名・変数名がそのレイヤーの責務を正しく反映しているか
- **フレームワークの機能活用**: フレームワークが提供する仕組みで実現できることを自前で実装していないか
- **意図外の変化がないか**: 手順 5 で宣言した意図の外に、挙動・副作用の変化が生じていないか。変更対象の旧コード/旧仕様が扱っていたケース（正常系・エラーパス・nil 等の境界）を列挙し、新コードで各ケースが同等に扱われるか確認する（意図的に変える場合は手順 5 で宣言されているはず）。

  **帯域外シグナルを変える場合はリポジトリ全体を見る**: 例外の送出、グローバル/静的状態の変更、イベント・通知の発行、ライフサイクルフックの発動、メタプログラミングによる動的挙動の変更など、**呼び出し元ではなく別の箇所で受信・観測される信号**を変える場合、受信側は immediate caller に現れない。この種の信号を変える時は受信側（例外なら捕捉箇所、イベントなら購読箇所、等）をリポジトリ全体で検索し、更新または互換性を確保する。戻り値や引数のように呼び出し元で完結する契約は、通常どおり呼び出し元の更新で足りる。

  **識別子をリネームした場合はリポジトリ全体を grep する**: クラス名・モジュール名・ファイルパス・関数名・定数名等を変更した場合、型チェッカーやテストでは捕捉できない参照（文字列リテラル、コメント、テストフィクスチャ名、設定ファイル、動的ロードのパス、`miko/` 配下の design docs・proposals・harae、README やその他 markdown、CI 設定など）が残る。`grep` でリポジトリ全体から旧名を洗い出し、ヒットごとに以下を判断する:

  - 同名の別概念ではないか（語境界・文脈の一致）
  - 過去の経緯記録（解決済み harae 項目・廃止 proposal・過去の commit log を引いた docs 等）として旧名のまま残すべきか
  - リネーム結果が文脈上正しい表記か（複合語・派生語の扱い）

  機械的な一括置換はしない。判断に迷うヒットは、置換せずに主さまに提示して確認する。

問題を発見したら修正する。

**セルフレビュー完了後、`/simplify` を実行する。**

### 7. デザインドキュメントの更新

**読み込み:**
- `.miko/guides/business_rules_guide.md` — ルールの書き方、実装マッピングの形式
- `.miko/examples/business_rules.md` — 品質基準
- `.miko/examples/high_level_design.md` — 品質基準

**方針:** 実際に変わったものだけを更新する。変わっていない部分には触らない。識別子リネームに伴う docs 内の旧名追従は、手順 6 のセルフレビュー（リポジトリ全体 grep）で実施済みの前提。

#### business_rules.md の BR 本文更新

- proposal がある場合のみ: proposal の新設・改訂・廃止を BR 本文に適用する
- proposal がない場合: BR 本文は一切触らない（スコープガード (d) で除外済みのはず）

#### 実装マッピングの横断更新（Sonnet サブエージェント）

実装の変更がどの cap の実装マッピングに波及したかを機械的にチェック・更新する。

まず主さまにベースブランチ名を確認する（例: 「このブランチのベースブランチ（マージ先）を教えてくださいませ」）。自動推定はしない。

Sonnet サブエージェント（Agent ツール、model: "sonnet"）を起動して実行する。

サブエージェントに渡す情報:
- `.miko/guides/cross_capability_mapping_guide.md` のパス — 「このファイルを読み、指示に従え」と指示する
- メインケイパビリティ名 — proposal がある場合のみ渡す（proposal がない場合は渡さず、全 cap スキャンにする）
- 主さまに確認したベースブランチ名

#### high_level_design.md の更新

- 構造の描写（コンポーネント配置、処理パイプライン等）が変わった場合のみ更新
- 識別子リネームの追従は上記「識別子リネーム時の横断追従」で実施済み。本セクションでは構造変更の有無のみ判断する
- 「設計上の特徴」セクションも、スコープガードに通った範囲であれば内容に応じて更新してよい（スコープガードが複雑な設計判断の変更を先に弾いている前提）

#### harae.md への転記

- proposal に「祓え検証」セクションがある場合、その内容を `miko/<capability>/harae.md` に適用する。harae.md のフォーマットは `.miko/guides/harae_format_guide.md` に従う
- **既存指摘のステータス変更**: 祓え検証に記載されたステータス変更を harae.md の該当エントリに反映する。指摘一覧テーブルのステータスセル更新・打ち消し線付与、詳細セクションのステータスと解決フィールドの更新を行う
- **新規指摘の追加**: 祓え検証の新規指摘を harae.md に追加する。番号は既存の最大番号の続きから採番する。ステータス・解決フィールドが省略されている場合は `open` / `—` として補完する
- proposal がない、または祓え検証セクションがない場合はスキップ

#### 親プロポーザルの進捗更新

- proposal の先頭に `<sub-proposal@{親のmiko相対パス}>` マーカーがある場合、`miko/{パス}` で親プロポーザルを開き、該当フェーズのチェックボックスにチェックを入れる（`- [ ]` → `- [x]`）
- 該当しなければスキップ

**更新内容を提示し、主さまの確認を取る。**

### 8. 完了報告

以下をサマリーテーブルで提示する:

- 入力種別（proposal / cap / 変更指示）と入力内容
- 変更ファイル数・行数
- テスト数・実行結果
- セルフレビュー修正数
- BR 本文変更件数（該当なしなら 0）
- 実装マッピング更新件数（cap 別、該当なしなら 0）
- HLD 更新有無
- harae 転記件数（該当なしなら 0）

テストの実行を依頼する。
